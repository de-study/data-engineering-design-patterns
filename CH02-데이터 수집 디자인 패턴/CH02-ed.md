## **Chapter 2. 데이터 인젝션 설계 패턴**

### **2.1 전체 적재 (Full Load)**

#### **패턴 #01: 전체 로더 (Full Loader)**

- **문제 상황**
    - 마지막 데이터 수집 이후 변경된 레코드 감지에 대한 정보를 알 수 없는 경우
    - 소스 데이터에 'updated_at' 컬럼이 없어서 레코드가 언제 업데이트 됐는지 알 수 없음
        
    - 데이터가 작아서(100만 건 이하) 그냥 다 가져오는 게 머리 안 아픈 경우
        
- **해결책**
    - 매번 전체를 읽어서 타겟 테이블에 덮어쓰기(`Overwrite`)
        
- **사수의 조언 (엔지니어 독백)**
    
    > "야, 처음엔 이게 제일 편해 보여도 제일 위험해. 만약 네가 적재하는 동안 분석가가 쿼리를 날리면 어떻게 될까? 테이블이 비어있거나 데이터가 반토막 난 걸 보게 되겠지? 그래서 실무에선 '스테이징 테이블'에 먼저 다 붓고, 마지막에 **View나 Alias**만 슥 바꿔치기하는 식으로 처리해야 욕 안 먹어. 그리고 데이터 100만 건 넘어가기 시작하면 매일 아침 파이프라인 느려진다고 알람 올 테니까 미리 증분 적재 준비해라."
    
- **사용 앱**: **AWS Glue, Python (on EC2/Lambda)**
- **활용 상황**: "기준 정보" 데이터 수집. 예를 들어 서비스 내 '상품 카테고리 코드', '법정동 코드'처럼 양은 적은데 변경 이력 추적 컬럼이 없는 데이터를 DB에서 S3로 긁어올 때 써.
    
- **사수의 조언 (엔지니어 독백)**:
    > "Glue 썼다고 안심하지 마. 데이터 긁어오는 동안 소스 DB에 락(Lock) 걸어서 서비스 마비시키는 신입들 꼭 있어. `SELECT *` 날리기 전에 DB 부하 꼭 확인하고, 가능하면 읽기 전용 복제본(Read Replica)에서 가져와."

### **2.2 증분 적재 (Incremental Load)**

#### **패턴 #02: 증분 로더 (Incremental Loader)**

- **문제 상황**
    
    - 데이터가 너무 많아서 매번 다 가져오면 비용이랑 시간 감당 안 됨
        
- **해결책**
    
    - **델타 컬럼**: `updated_at` 기준 필터링
        
    - **파티션**: 오늘 날짜 폴더(`date=2026-05-09`)만 콕 집어서 로드
        
- **사수의 조언 (엔지니어 독백)**
    
    > "이건 '누락'과의 전쟁이야. 소스 DB에서 누가 데이터를 몰래 수정했는데 `updated_at` 안 바꿔놨다? 그럼 그 데이터는 평생 우리 쪽에 안 들어와. 또 제일 미치는 건 **Late Data**야. 10시 데이터를 11시에 가져가는데, 네트워크 밀려서 11시 1분에 들어온 10시 데이터는? 그냥 버려지는 거지. 그래서 증분 로더 짤 때는 항상 '앞뒤로 1시간 정도 겹쳐서 읽기(Windowing)'나 재처리 로직을 염두에 둬야 해."
    
- **사용 앱**: **Apache Airflow, Apache Spark**
    
- **활용 상황**: 배치(Batch) 작업의 정석이야. Airflow로 스케줄링하고 Spark로 어제 들어온 로그만 필터링해서 적재할 때 써. 주로 서비스 로그나 일별 정산 데이터 수집에 쓰지.
    
- **사수의 조언 (엔지니어 독백)**:
    
    > "Airflow의 `execution_date` 개념을 잘 이해해야 해. 오늘 도는 배치는 어제 데이터를 가져오는 경우가 많거든. 그리고 **멱등성(Idempotency)**! 똑같은 작업을 두 번 돌려도 데이터가 중복으로 쌓이지 않게 `Overwrite` 파티션 로직을 짜야 나중에 재처리할 때 고생 안 한다."
    

#### **패턴 #03: 변경 데이터 캡처 (Change Data Capture)**

- **문제 상황**
    
    - "30초 안에 데이터 반영해 주세요" 같은 미친 요구사항이 들어옴
        
    - 삭제된 데이터(Hard Delete)까지 완벽하게 추적해야 함
        
- **해결책**
    
    - DB의 **커밋 로그**를 직접 빨아들여서 실시간 스트리밍
        
- **사수의 조언 (엔지니어 독백)**
    
    > "CDC는 데이터 엔지니어링의 꽃이지만, 운영은 가시밭길이야. 소스 DB 사장님이 갑자기 인덱스 바꾼다고 DB 내리면? 로그 읽던 Debezium도 같이 뻗어. 그리고 실시간이라고 좋아만 하지 마라. 분석 환경에선 1초 전 데이터랑 지금 데이터가 다르면 오히려 분석가가 헷갈려 할 수도 있어. 꼭 필요한 곳에만 써야 해."

- **사용 앱**: **Debezium, Apache Kafka**
    
- **활용 상황**: 서비스 DB(MySQL, PostgreSQL 등)의 변경 사항을 1초라도 빨리 분석계로 넘겨야 할 때. 혹은 '주문 취소'처럼 기존 데이터의 상태가 바뀌는 걸 즉시 감지해야 할 때 필수야.
    
- **사수의 조언 (엔지니어 독백)**:
    
    > "Debezium 설정은 진짜 예민해. DB의 Binlog 보관 주기랑 Kafka 토픽의 Retention 기간을 잘 맞춰야 해. 안 그러면 중간에 로그가 잘려서 데이터 구멍 난다? 그거 메우려면 전체 적재 다시 해야 하니까 정신 바짝 차려."

### **2.3 복제 (Replication)**

#### **패턴 #04: 패스스루 복제기 (Passthrough Replicator)**

- **문제 상황**
    
    - 운영 DB 데이터를 개발 환경으로 똑같이 복사해야 할 때
        
- **해결책**
    
    - 데이터 포맷 변환 없이 원본 그대로 바이트 단위 복사
        
- **사수의 조언 (엔지니어 독백)**
    > "데이터 옮길 때 '파싱(Parsing)'은 독이야. JSON을 굳이 읽어서 다시 저장하지 말고, 그냥 통째로 긁어다 옮겨. 왜냐고? 네가 숫자를 읽는 순간 반올림 오차가 생길 수도 있고, 날짜 포맷이 살짝 뒤틀릴 수도 있거든. 원본 데이터의 **순결함**을 지켜주는 게 복제의 핵심이야."
    
- **사용 앱**: **AWS DataSync, S3 Replication**
    
- **활용 상황**: 운영 환경 S3에 쌓인 이미지나 로그 파일을 분석 환경 버킷으로 똑같이 옮길 때 써. 코드 한 줄 안 짜고 클라우드 설정만으로 끝내는 게 베스트야.
    
- **사수의 조언 (엔지니어 독백)**:
    > "직접 파이썬 코드로 파일 복사하려고 하지 마. 데이터 유실 체크까지 다 해주는 클라우드 네이티브 서비스 쓰는 게 최고야. 대신 비용 옵션 잘 봐라. 리전 간 전송(Inter-region transfer) 비용 무시했다가는 다음 달 부장님 호출받는다."

#### **패턴 #05: 변환 복제기 (Transformation Replicator)**

- **문제 상황**
    
    - 복제는 해야 하는데, 그 안에 유출되면 쇠고랑 차는 개인정보가 있을 때
        
- **해결책**
    
    - 중간에 개인정보만 쏙 빼거나 가명화 처리해서 복제
        
- **사수의 조언 (엔지니어 독백)**
    
    > "야, 이건 네 목숨줄이야. 운영 데이터를 스테이징 환경으로 옮기는데 고객 전화번호가 그대로 들어갔다? 그럼 그날로 뉴스 나오는 거야. 컬럼 이름이 `phone`이면 무조건 버리게 설계해. 근데 가끔 소스 쪽에서 컬럼 이름을 `user_contact`로 바꿔버리면? 그래서 우리는 '민감 정보 태깅' 자동화를 계속 공부해야 하는 거야."

- **사용 앱**: **AWS Glue ETL, Spark**
    
- **활용 상황**: 운영 데이터를 외부 분석 플랫폼에 제공하거나 테스트 환경으로 옮길 때. 주민번호, 전화번호를 `SHA-256`으로 해싱하거나 아예 드랍(Drop)시키는 전처리가 필요할 때 써.
    
- **사수의 조언 (엔지니어 독백)**:
    
    > "복제 중간에 데이터 형 변환(Type casting) 함부로 하지 마. 원본은 숫자인데 네가 실수로 문자열로 받아서 복제하면 나중에 분석가가 덧셈이 안 된다고 울면서 찾아온다. 개인정보 처리 규칙은 문서로 명확히 남겨두고."

### **2.4 데이터 컴팩션 (Data Compaction)**

#### **패턴 #06: 컴팩터 (Compactor)**

- **문제 상황**
    
    - 실시간으로 쌓다 보니 1KB짜리 파일이 수만 개 생겨서 쿼리가 멈춤
        
- **해결책**
    
    - 작은 파일들을 모아서 큼직한 파일(1GB 단위)로 합치기
        
- **사수의 조언 (엔지니어 독백)**
    
    > "이걸 안 하면 나중에 분석가가 너한테 달려온다. '왜 쿼리가 1시간 동안 안 끝나요?'라고. 파일 리스트만 읽다가 시간 다 쓰는 거거든. 주기적으로 **OPTIMIZE** 돌려줘야 해. 근데 조심할 건, 컴팩션 하면 예전 파일들이 찌꺼기로 남거든? **VACUUM** 안 돌려주면 클라우드 저장소 비용 폭탄 맞으니까 세트로 기억해라."
    
- **사용 앱**: **Databricks, EMR (Spark)**
    
- **활용 상황**: 스트리밍으로 들어온 데이터가 S3에 수만 개의 작은 파일로 쪼개져 있을 때. 쿼리 속도가 기어갈 때 Spark의 `repartition`이나 Delta Lake의 `OPTIMIZE`를 써서 합쳐줘.
    
- **사수의 조언 (엔지니어 독백)**:
    
    > "컴팩션은 공짜가 아냐. 파일을 다시 읽어서 쓰는 거라 컴퓨팅 비용이 꽤 나가. 실시간성이 덜 중요한 데이터라면 하루에 한 번만 모아서 돌려. 그리고 **VACUUM** 잊지 마. 예전 파일 안 지우면 스토리지 비용만 계속 늘어난다."
---

### **2.5 데이터 준비 (Data Readiness)**

#### **패턴 #07: 준비 마커 (Readiness Marker)**

- **문제 상황**
    
    - 데이터 아직 다 안 들어왔는데 다음 작업이 시작돼서 숫자가 틀림
        
- **해결책**
    
    - 적재 완료 직후 `_SUCCESS` 파일 생성해서 신호 주기
        
- **사수의 조언 (엔지니어 독백)**
    
    > "이건 데이터 엔지니어 사이의 약속이야. 네가 데이터를 다 쌓았다고 생각해도 네트워크는 아직 파일을 쓰고 있을 수 있어. 확실하게 '나 다 했어!'라고 깃발을 꽂아줘야 뒤에 있는 사람이 안심하고 먹지. Airflow 쓸 때 **Sensor** 달아놓는 거 잊지 말고."
    
- **사용 앱**: **Airflow Sensors, AWS SNS/SQS**
    
- **활용 상황**: 업스트림 팀이 데이터를 다 썼는지 확인할 때. S3에 `_SUCCESS` 파일이 생기면 SNS로 알람을 쏘고, 그걸 받은 후속 DAG가 실행되게 설계해.
    
- **사수의 조언 (엔지니어 독백)**:
    
    > "팀 간의 '신뢰'는 파일 하나에서 시작돼. '이 정도 시간이면 다 들어왔겠지?'라는 추측이 데이터 사고의 90%야. 무조건 마커 파일을 확인해. 만약 소스 팀에서 마커를 안 만들어주면? 네가 데이터 건수를 체크하는 커스텀 센서라도 만들어서 검증해야 해."

---

### **2.6 이벤트 주도 (Event Driven)**

#### **패턴 #08: 외부 트리거 (External Trigger)**

- **문제 상황**
    
    - 데이터가 언제 들어올지 모르는데 하루 종일 켜놓자니 돈 아까울 때
        
- **해결책**
    
    - S3에 파일 들어오는 순간 파이프라인 가동 (Push 방식)
        
- **사수의 조언 (엔지니어 독백)**
    
    > "가성비는 최고지. 근데 트리거가 실패하면 어떻게 될까? 이벤트는 날아갔는데 파이프라인은 안 돌았어. 그럼 그 데이터는 영영 잃어버리는 거야. 그래서 **Dead Letter Queue** 같은 걸 만들어서 실패한 이벤트는 따로 모아둬야 해. 안 그러면 나중에 소명하느라 밤샌다."

- **사용 앱**: **AWS Lambda, S3 Event Notifications**
    
- **활용 상황**: 불규칙하게 들어오는 파일 처리. 예를 들어 고객사에서 언제 보낼지 모르는 CSV 파일을 받자마자 바로 DB에 밀어 넣어야 할 때 아주 유용해.
    
- **사수의 조언 (엔지니어 독백)**:
    
    > "람다는 가볍지만 무책임해. 트리거가 실패했을 때 자동으로 재시도 안 해주는 경우가 많거든. 실패한 이벤트는 **SQS(Queue)**에 담아두거나 **CloudWatch Alarm**을 걸어서 네가 직접 확인할 수 있는 장치를 꼭 만들어야 해."
    

### 2.7 정리 및 기타
**정리하자면:**

- **DB 로그**는 **CDC(Debezium)**
    
- **대량의 배치 작업**은 **Airflow + Spark**
    
- **파일 이벤트**는 **Lambda**
    
- **정리는** **Spark(Compactor)**


#### Q.s3파일 업로드 되는거 감지는 어떤 서비스로하나?
 **1. AWS Lambda (가장 대중적인 조합)**

- **작동 방식**: S3 버킷 설정에서 '이벤트 알림(Event Notifications)'을 켜고, 대상(Destination)을 **Lambda**로 지정해. 파일이 들어오자마자 람다 함수가 깨어나서 코드를 실행해.
    
- **실무 예시**: CSV 파일이 S3에 올라오자마자 파이썬 코드가 돌면서 DB에 데이터를 집어넣거나, 이미지 파일의 썸네일을 만들 때 써.
    
- **사수의 조언**:
    
    > "제일 편하긴 한데, 파일이 초당 수천 개씩 미친 듯이 들어오면 람다 호출 비용이 폭탄처럼 터질 수 있어. 그리고 람다는 실행 시간이 15분 제한이니까, 너무 무거운 작업(대용량 압축 풀기 등)은 시키지 마라."
    

 **2. AWS SQS (메시지 큐 - 안정성 중심)**

- **작동 방식**: S3 이벤트 알림을 Lambda로 바로 쏘는 게 아니라, **SQS**라는 대기열(Queue)에 먼저 쌓아두는 방식이야.
    
- **실무 예시**: 파일이 불규칙하게 대량으로 들어올 때 써. 람다가 한꺼번에 수천 개 뜨는 걸 방지하고, 우리가 원하는 속도만큼만 큐에서 꺼내서 처리하고 싶을 때(Throttling) 필수야.
    
- **사수의 조언**:
    
    > "서버가 잠시 죽어도 이벤트가 사라지지 않게 보호해주는 '완충 지대'라고 생각하면 돼. 처리하다 실패해도 큐에 남아있으니까 나중에 재처리(Retry)하기도 훨씬 쉽지."
    

 **3. Amazon EventBridge (복잡한 라우팅용)**

- **작동 방식**: S3의 모든 이벤트를 **EventBridge 버스**로 보내고, 거기서 규칙(Rule)을 정해 여러 곳으로 뿌려주는 방식이야.
    
- **실무 예시**: "파일이 업로드되면 람다도 실행하고, 동시에 운영팀에 이메일 알림도 보내고, 로그도 남겨라"처럼 한 가지 사건으로 **여러 액션**을 취해야 할 때 써.
    
- **사수의 조언**:
    
    > "단순히 파일 하나 처리할 거면 람다로 충분하지만, 기업급 시스템에서 이벤트 표준을 맞추거나 다른 서비스랑 연동이 많아지면 결국 이놈을 쓰게 될 거야."
    

---
[Summary]

- "그냥 바로 처리하고 싶다"** → S3 + Lambda
    
- "실패하면 안 되고, 차근차근 처리하고 싶다" → S3 + SQS + Lambda
    
- "이벤트 하나로 여러 서비스를 호출하고 싶다" → S3 + EventBridge  



#### Q2.airflow sensing 작동방식은? 
 **Airflow Sensor의 핵심 개념**

- **작동 방식**: 정해진 시간 간격(`poke_interval`)마다 대상(S3, DB, HTTP 등)을 찔러봐서(Poke) 결과가 `True`이면 다음 태스크로 넘어가고, `False`이면 다시 기다려.
    
- **사용 앱**: `S3KeySensor`, `ExternalTaskSensor`, `SqlSensor`, `HttpSensor` 등
    

---

 **패턴별 Sensing 활용 예시 및 사수의 조언**

#### **1. S3KeySensor (파일 감지)**

- **활용**: 타 팀에서 S3에 `_SUCCESS` 파일을 생성해주기로 했을 때 사용.
    
- **상황**: "데이터가 다 들어왔나?" 확인 후 Spark 배치 작업을 돌릴 때.
    
- **사수의 조언**:
    
    > "야, 센서 쓸 때 제일 조심해야 하는 게 **Worker 점유**야. 기본 설정(`mode='poke'`)으로 두면 센서가 기다리는 동안 워커 슬롯 하나를 꽉 잡고 안 놔줘. 기다리는 센서가 많아지면 진짜 실행되어야 할 작업들이 줄을 서서 대기하는 대참사가 나지. 무조건 **`mode='reschedule'`**로 설정해. 그래야 체크 안 할 때는 슬롯을 반환하고 잠시 잠들었다가 다시 깨어난다."
    

#### **2. ExternalTaskSensor (다른 DAG 완료 감지)**

- **활용**: 상위(Upstream) 파이프라인 DAG가 성공적으로 끝났는지 확인할 때 사용.
    
- **상황**: 유저 로그 인젝션 DAG가 끝나야 통계 계산 DAG가 돌 수 있는 경우.
    
- **사수의 조언**:
    
    > "이건 두 DAG의 스케줄 주기가 정확히 맞아야 해. 상위 DAG는 매일 01:00에 도는데 하위 센서는 02:00에 찾으려고 하면 못 찾고 타임아웃 난다. `execution_delta` 설정을 통해 시차를 맞추는 법을 꼭 익혀둬야 해."
    

#### **3. SqlSensor (데이터 상태 감지)**

- **활용**: DB에 특정 조건의 레코드가 들어왔는지 확인할 때 사용.
    
- **상황**: "배치 정산용 데이터가 1,000건 이상 쌓였는가?" 혹은 "마스터 테이블 업데이트 완료 플래그가 들어왔는가?" 체크.
    
- **사수의 조언**:
    
    > "DB에 직접 날리는 센서는 쿼리가 가벼워야 해. 1분마다 `SELECT COUNT(*)` 같은 무거운 쿼리를 날리면 센서 때문에 운영 DB 성능이 박살 날 수 있어. 가능하면 상태값만 체크하는 가벼운 쿼리를 써라."
    

---

 **Sensor 사용 시 절대 주의사항 (신입 필독)**

1. **Timeout 설정 (생명줄)**:
    - 센서에 `timeout`을 안 걸면, 조건이 충족 안 될 경우 영원히(최대 7일) 기다리며 리소스를 좀먹어. 보통 1시간이나 2시간 정도로 적절한 타임아웃을 걸고, 실패 시 알람을 받게 설계해.
        
2. **Soft Fail**:
    - 데이터가 안 들어왔다고 파이프라인 전체를 '실패(Failed)'로 만들고 싶지 않을 때가 있어. 그럴 땐 `soft_fail=True`를 주면 실패가 아니라 '건너뜀(Skipped)' 처리가 돼서 깔끔해.
        
3. **Poke Interval**:
    - 너무 자주 찌르면(`poke_interval=5`) 리소스 낭비고, 너무 가끔 찌르면(`300`) 전체 파이프라인이 느려져. 데이터의 중요도에 따라 30초~2분 사이에서 결정하는 게 보통이야.
