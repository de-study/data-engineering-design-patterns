## 패턴 #42: 제자리 덮어쓰기

### 1\. 개요 및 문제 정의

-   **개요**: 수직 파티셔너 패턴은 시간과 컴퓨팅 자원이 충분할 때 유용하지만, 그렇지 못한 상황에서는 검증된 덮어쓰기 전략인 '제자리 덮어쓰기' 패턴이 대안이 된다.
-   **문제**: 테라바이트 규모의 데이터가 시간 기반 수평 파티션에 저장된 레거시 시스템을 넘겨받았으나, 개인 데이터 관리 전략이 아직 정의되지 않은 상황이다. 이 프로젝트는 조직 내에서 널리 사용되고 있으며, 정부의 새로운 개인정보 보호 규정에 따라 사용자가 요청할 때 개인 데이터를 즉시 제거해야 하는 요구사항을 충족해야 한다.

### 2\. 해결책 및 구현 방식

현재 아키텍처 리팩터링을 위한 자원이 부족하므로 **제자리 덮어쓰기(In-Place Overwriter)** 패턴을 적용한다. 구현은 데이터 스토리지 기술에 따라 다르게 나타난다.

-   **기본 삭제 기능을 지원하는 경우 (In-place operation)**
    -   WHERE 조건을 통해 제거할 엔티티를 대상으로 DELETE 문을 실행한다.
    -   아파치 아이스버그나 델타 레이크 같은 오픈 테이블 파일 형식은 잘못된 작업 시 이전 버전으로 복원하는 '타임 트래블' 기능을 제공한다. 다만, 개인 데이터를 삭제하더라도 제거된 레코드를 저장하는 데이터 블록을 회수하지 않으면 데이터가 그대로 남게 되므로 주의해야 한다.
    -   **삭제 벡터(Deletion Vector)**: 테이블 파일 형식에서 삭제를 관리하는 방식은 두 가지이다.
        1.  제거된 레코드를 식별하고 쓰기 풋프린트를 줄이기 위해 이를 더 작은 측면 파일(side file)에 쓰는 방식이며, 이를 **삭제 벡터**라고 한다. 컨슈머가 조회 시점에 삭제된 레코드를 직접 제거해야 한다.
        2.  반대의 방식은 컨슈머가 바로 사용할 수 있도록 제거된 항목을 제외한 모든 데이터를 파일에 새로 쓰는 방식이며, 상대적으로 더 쓰기 집약적이다.
-   **기본 삭제 기능을 제공하지 않는 경우 (JSON, CSV 등 원시 파일 형식)**
    -   전체 데이터셋을 처리하고 제거된 사용자의 레코드를 필터링하는 시뮬레이션 잡을 실행해야 한다. 기존 데이터를 필터링된 데이터셋으로 교체하는 것은 컴퓨팅 집약적인 작업이다.
    -   잡이 실패하거나 재시도될 때 발생할 수 있는 데이터 손실을 방지하기 위해 데이터셋을 직접 교체해서는 안 된다.
    -   **대안(스테이징 영역 활용)**: 임시 저장 영역인 스테이징 영역(staging area)에 데이터를 먼저 쓰고, 데이터 제거가 성공적으로 완료된 후에만 기존 공개 데이터셋을 덮어쓰는 **데이터 승격 잡**을 실행한다. 승격 잡이 실패하더라도 이미 스테이징 영역에 계산이 완료되어 있으므로 데이터 손실로 이어지지 않는다. 
        -   **파이스파크(PySpark) 및 AWS CLI 구현 예시**:

```
# 예제 7-5: 파이스파크로 플랫 파일에서 레코드 제거
input_raw_data = spark_session.read.text(get_input_table_dir())
df_w_user_column = input_raw_data.withColumn(
    'user', F.from_json('value', 'user_id STRING')
)
user_id = '139621130423168_029fba78-15dc-4944-9f65-00636566f75b'
to_save = df_w_user_column.filter(f"user.user_id != '{user_id}'").select('value')
to_save.write.mode('overwrite').format('text').save(get_staging_table_dir())
```

텍스트 API를 사용하여 데이터셋 변경을 최소화하고, 공간 절약을 위해 필터에 필요한 user\_id 속성만 추출한다. 아파치 스파크는 분산형이고 트랜잭션이 없는 처리 계층이므로 최종 위치에 바로 덮어쓰지 않고, 스테이징 위치에 생성한 뒤 rename-like 명령으로 최종 출력에 승격시킨다.

```
# 예제 7-6: AWS CLI를 이용한 최종 위치로의 데이터셋 승격
aws s3 rm ${BUCKET}/output --recursive
aws s3 mv ${BUCKET}/staging ${BUCKET}/output --recursive
```

-   **컴팩션 가능한 데이터 스토어 (아파치 카프카 토픽 등)**
    -   컴팩션 전략이 활성화되고 키 기반 레코드가 작성된 경우에도 작동한다. 토픽 레코드를 조회한 뒤, 제거된 엔티티에 속하면 레코드 키와 함께 **null 페이로드**를 토픽으로 보낸다. 컴팩션 프로세스가 이를 인식하여 이전 레코드를 제거하고 빈 삭제 마커를 저장한다. (키 기반 레코드와 토픽에 정의된 컴팩션 전략 필수)

> NOTE 보존 기간  
>   
> 데이터 보존 기간이 제거 작업에 걸리는 지연 시간보다 짧다면 자동으로 데이터가 제거되므로 이를 컴팩션 전략으로 고려할 수 있다. 단, 최종 해결책으로 도입하기 전에 최고 데이터 책임자(CDO)와 법무 부서의 확인을 거쳐야 한다.

### 3\. 패턴의 결과 및 한계

-   **I/O 오버헤드**: 많은 조회 및 쓰기 작업을 수행하므로 심각한 I/O 오버헤드가 발생한다. 시간이 흐름에 따라 스토리지 공간이 거의 두 배로 커지고 처리량 증가로 이어진다.
    -   _완화 방안_: 아파치 파케이(Apache Parquet)처럼 메타데이터 계층에 데이터 블록별 레코드 통계를 저장하는 형식을 사용하면, 쿼리 엔진이 통계를 분석해 제거 대상이 없는 데이터 블록의 읽기 작업을 건너뛸 수 있어 오버헤드가 줄어든다.
-   **비용**: 모든 데이터를 조회해야 하므로 수직 파티셔너 패턴보다 비용이 많이 든다. (예: 엔티티 하나에 2,000개 행이 있다면 제자리 덮어쓰기는 2,000개 행 전체에 영향을 준다.)
    -   _완화 방안_: 제거 요청마다 파이프라인을 실행하는 대신, 요청을 그룹화하여 하나의 파이프라인으로 처리한다.
-   **불가능한 롤백(CAUTION)**: 데이터 제거 잡에 버그가 있어 원본 데이터를 이미 덮어써 버렸다면 롤백 및 재처리가 불가능하다. 이를 완화하기 위해 프록시 패턴을 적용하거나, 주요 객체 스토어(S3, Azure Storage, GCS 등)의 인프라 차원에서 데이터 버전 관리를 활성화해야 한다.

## 패턴 #43: 테이블 접근자 

### 1\. 개요 및 문제 정의

-   **개요**: 컴플라이언스 준수 외에도 보안을 위해 승인된 사용자만 중요한 데이터 섹션에 접근할 수 있도록 세밀한 접근 제어가 필요하다. 개인 데이터 비공개도 중요하지만, 데이터 자체는 조직의 가장 큰 경쟁 자산이기 때문이다.
-   **문제**: 이전의 HDFS/Hive 워크로드를 클라우드 데이터 웨어하우스로 마이그레이션한 후 보안 접근 정책을 구현해야 한다. 사용자 및 그룹별 테이블 접근 관리는 기본 기능으로 쉽게 충족되지만, 이해관계자들은 사용자가 특정 테이블에 접근 권한이 있더라도 **모든 컬럼과 레코드(row)를 읽을 수 없도록 저수준 자원 제어 메커니즘**을 요구하고 있다.

### 2\. 해결책 및 구현 방식

**테이블에 대한 세밀한 접근자(Fine-Grained Accessor for Tables)** 패턴을 통해 컬럼 및 레코드 수준의 저수준 데이터 접근 문제를 해결한다.

#### \[1\] 컬럼 기반 접근 제어 구현

1.  **GRANT 연산자 활용 (아마존 레드시프트, PostgreSQL)**
    -   권한이 부여된 작업 범위 내에서 컬럼을 정의한다. 권한이 없는 컬럼을 포함해 SELECT \* 같은 쿼리를 실행하면 데이터베이스가 ERROR: permission denied for table 오류를 반환한다.

```
# 예제 7-7 & 7-10: GRANT 문을 통한 특정 컬럼 접근 권한 부여
GRANT SELECT(col_A, col_B) ON my_table TO some_user;
GRANT SELECT(id, login, registered_datetime) ON dedp.users TO user_a;
```

2. **정책 태그 활용 (GCP 빅쿼리)**

-   데이터 카탈로그에서 정책 태그를 생성하고, 이를 빅쿼리 테이블의 보호된 컬럼에 할당한다. 이후 각 정책 태그에 세밀한 독자(fine-grained reader) 역할을 할당하여 접근 권한을 부여한다.

3\. **데이터 마스킹 기능 활용 (데이터브릭스 유니티 카탈로그, 스노우플레이크)**

-   사용자가 컬럼을 볼 수는 있지만 권한이 없으면 내용이 숨겨지도록 처리한다. 사용자가 특정 그룹에 속할 때만 원본을 보여주는 마스킹 함수를 적용하는 방식이다.

```
# 예제 7-8: 데이터브릭스의 컬럼 마스킹 함수 예제
CREATE FUNCTION ip_mask(ip STRING)
  RETURN CASE WHEN is_member('engineers') THEN ip ELSE ' . . . ' END;

CREATE TABLE visits (
  visit_id STRING,
  ip STRING MASK ip_mask
);
```

#### \[2\] 레코드(Row) 수준 접근 제어 구현

-   실행된 쿼리의 WHERE 조건에 동적 함수를 즉석에서 추가하는 방식에 의존한다. 엔진마다 부르는 이름이 다르다.
    -   데이터브릭스: **ROW FILTER**
    -   아마존 레드시프트: **Row-Level Security**
    -   GCP 빅쿼리 & 스노우플레이크: **레코드(row) 접근 정책**
    -   PostgreSQL: **ROW LEVEL SECURITY** (ALTER TABLE 테이블명 ENABLE ROW LEVEL SECURITY; 후 CREATE POLICY 정책 생성)
-   **PostgreSQL 예시**:

```
# 예제 7-11: PostgreSQL 레코드 수준 접근 정책
ALTER TABLE dedp.users ENABLE ROW LEVEL SECURITY;
CREATE POLICY user_row_access ON dedp.users USING (login = current_user);
```

-   기본 지원이 없는 경우 (시뮬레이션): 접근 조건이 동적으로 반영되는 뷰(View)를 생성하여 테이블을 노출한다. 사용자가 쿼리를 실행할 때 해당 사용자가 소유한 레코드만 반환되므로, 사용자마다 전용 버전의 뷰를 갖는 효과를 낸다.

```
# 예제 7-9: 뷰를 사용한 레코드 수준 접근 시뮬레이션
CREATE VIEW users_blogs AS
SELECT ... FROM blogs WHERE table.blog_author = current_user;
```

-   NoSQL 데이터베이스 (AWS DynamoDB) 사례:
    -   IAM 정책과 Condition 절의 dynamodb:LeadingKeys 속성을 조합하여 사용자가 자신의 user\_id 값으로 시작하는 레코드만 조회할 수 있도록 세분화된 접근 정책을 구현한다.

```
/* 예제 7-12: AWS DynamoDB 세분화된 접근 정책 */
{
  "Statement":[{
    "Sid": "...",
    "Effect": "Allow",
    "Action": [...],
    "Resource": ["arn:aws:dynamodb:us-west-1:12456789012:table/users"],
    "Condition": {
      "ForAllValues:StringEquals": {
        "dynamodb:LeadingKeys": ["${www.amazon.com:user_id}"]
      }
    }
  }]
}
```

### 3\. 패턴의 결과 및 제한 사항

-   **장점**: 데이터베이스에서 기본으로 지원하는 기능이므로 기존의 다른 패턴들에 비해 단점이 상대적으로 적다.
-   **레코드 수준 보안의 제한**: 대부분의 구현이 사용자 이름, 그룹, IP 등 연결 세션에서 직접 얻을 수 있는 속성들로만 조건을 구성할 수 있어 적용 범위가 제한적이다.
-   **데이터 유형의 제약**: 컬럼이 중첩 구조와 같은 복합 타입인 경우 간단한 컬럼 기반 접근 전략을 쓰기 어렵다. 이를 해결하려면 초기 구조에서 중첩을 해제(비중첩화)하여 별도 테이블로 노출하거나, 세밀한 권한을 지원하는 구조로 '데이터셋 구체화' 패턴을 사용해야 한다.
-   **쿼리 오버헤드**: 실행 시점에 SQL 함수가 동적으로 추가되므로 예상치 못한 지연(오버헤드)이 발생할 수 있다. 지연이 심할 경우 해당 그룹이 읽을 수 있는 데이터만 별도로 만드는 '데이터셋 구체화' 패턴으로 완화할 수 있으나, 이 역시 데이터 중복이나 다수 접근 그룹에 대한 거버넌스 제한이라는 부작용이 동반된다.

## 패턴 #44: 자원  접근자 

### 1\. 개요 및 문제 정의

-   **개요**: 접근 기반 패턴은 테이블 기반 데이터셋에서 잘 작동하지만, 데이터베이스가 데이터 엔지니어가 사용하는 유일한 데이터 스토어는 아니다. 클라우드 제공업체에서 관리하는 다양한 데이터 스토어 자원을 이용할 때 이 패턴이 적용된다.
-   **문제**: 보안 감사에서 클라우드 계정의 지나치게 광범위한 권한이 발견되었다. 발견된 위험은 하나의 데이터 처리 잡이 객체 스토어에서 사용 가능한 모든 데이터셋을 덮어쓸 수 있다는 가능성이다. 감사관은 시스템의 각 구성 요소에 필요한 최소 권한을 할당하여 데이터 처리 작업이 실제로 다루고 있는 데이터셋만 조작할 수 있도록 하는 최소 권한이라는 보안 모범 사례를 제시했다. 현재 클라우드 제공업체에서 이러한 최소 권한을 구현하기 위한 기술적 해결책을 찾고 있다.

### 2\. 해결책 및 구현 방식

AWS, Azure, GCP를 포함한 모든 주요 클라우드 제공업체는 자원에 대한 세밀한 접근자 패턴의 핵심인 **최소 권한 원칙**에 대한 구현을 제공한다. 클라우드 제공업체는 자원에 대한 접근을 제한하기 위해 크게 두 가지 전략을 사용한다.

#### \[1\] 자원 기반 접근 제어 (Resource-based Access Control)

-   **정의**: 자원 수준에서 직접 접근 범위를 정의하는 방식이다. 접근 제어 메커니즘이 각각의 자원에 직접 연결되며, 사용자 수준에 위치한다.
-   **구현 예시 (GCP GCS)**: GCP의 GCS 버킷에 대한 접근을 제어하려면 IAM 정책을 할당해야 한다.

```
# 예제 7-13: 테라폼에서 GCS를 위한 IAM 정책
data "google_iam_policy" "admin_access" {
  binding {
    role    = "roles/storage.admin"
    members = [
      "user:admingcs@waitingforcode.com",
    ]
  }
}

resource "google_storage_bucket_iam_policy" "policy" {
  bucket      = google_storage_bucket.default.name
  policy_data = data.google_iam_policy.admin_access.policy_data
}
```

#### \[2\] ID 기반 접근 제어 (ID-based Access Control)

-   **정의**: 자원 대신 ID 수준(가령 사람 또는 애플리케이션 사용자)에서 접근 권한이 직접 정의되는 방식이다.
-   **구현 예시 (AWS IAM 역할)**: AWS는 서비스에서 가정하는 IAM 역할로 이 모드를 지원한다. 아파치 스파크 EMR 작업이 키네시스 데이터 스트림에 접근할 때 읽기 및 쓰기 권한을 부여하는 방식을 보여준다.

```
# 예제 7-14: 테라폼을 사용한 AWS 키네시스 데이터 스트림에 대한 읽기 및 쓰기 접근
data "aws_iam_policy_document" "emr_assume_role" {
  statement {
    effect = "Allow"
    principals {
      type        = "Service"
      identifiers = ["elasticmapreduce.amazonaws.com"]
    }
    actions = ["sts:AssumeRole"]
  }
}

resource "aws_iam_role" "job_role" {
  name               = "visits-processor-role"
  assume_role_policy = data.aws_iam_policy_document.emr_assume_role.json
}

resource "aws_iam_policy" "visits_read_write_policy" {
  name   = "visits_rw"
  policy = jsonencode({
    Version   = "2012-10-17"
    Statement = [{
      Action   = ["kinesis:Get*", "kinesis:Describe*", "kinesis:List*", "kinesis:Put*"]
      Effect   = "Allow"
      Resource = ["arn:aws:kinesis:us-east-1:123456789012:streams/visits"]
    }]
  })
}

resource "aws_iam_role_policy_attachment" "policy_attachment" {
  role       = aws_iam_role.job_role.name
  policy_arn = aws_iam_policy.visits_read_write_policy.arn
}
```

#### \[3\] 세밀한 권한의 유연성과 태그 기반 접근 제어

-   세밀한 권한은 종종 매우 유연하다. 특정 자원뿐만 아니라, 동일한 접두사로 시작하는 자원 집합 또는 런타임 조건에 기반한 자원까지 대상으로 할 수 있다. 이는 인프라의 가장 낮은 수준에서 접근을 제어하기 때문에 이전 패턴의 레코드 수준 접근과 유사해 보인다.
-   **구현 예시 (AWS S3 태그 기반 접근 제어)**: 각 자원에 개별적으로 연결된 사용자 정의 메타데이터 속성(태그)에 의존하여 권한을 제어한다.

```
/* 예제 7-15: AWS S3에 대한 태그 기반 접근 정책 */
"Statement": [{
  "Effect": "Allow",
  "Action": "s3:PutObject",
  "Resource": "*",
  "Condition": {
    "ForAllValues:StringEquals": {"aws:TagKeys": ["${www.amazon.com:user_id}"]}
  }
}]
```

### 3\. 패턴의 결과 및 제한 사항 (보안 트레이드오프)

-   **장점**: 코드로서의 인프라(IaC) 도구나 사용자 정의 스크립트에 의존할 수 있으므로 기술적 관점에서 접근 정책을 정의하는 것이 어렵지 않아야 한다.
-   **원칙에 따른 보안 트레이드오프**: 최소 권한 원칙은 훌륭하지만 복잡한 환경에서 유지 관리하기 어려운 작은 접근 정책 정의를 많이 만들어낼 수 있다.
    -   _완화 방안_: 자원을 관리 가능한 크기로 유지하기 위해 와일드카드 기반 접근을 사용할 수 있다. 예를 들어 visits\* 같은 접두사를 사용하면 이 접두사로 시작하는 모든 자원에 접근 정책이 적용된다. 다만, 미래에 생성될 모든 visits 접두사 자원에 사용자가 접근할 권한이 있어야 한다고 장담할 수 없으므로 최소 권한 원칙을 위반할 수 있다. 이 때문에 보안 부서와 간소화 전략을 논의해야 한다.
-   **복잡성**: 같은 프로젝트에서 자원 기반 방식과 ID 기반 접근 방식을 모두 사용하면 의도치 않게 시스템의 복잡성을 증가시킬 수 있다. 가급적 더 많은 사용 사례를 다루는 해결책 하나를 선호하는 것이 좋다.
-   **할당량 제한**: 여느 클라우드 자원과 마찬가지로 접근 정책에도 제한이 있다.
    -   AWS IAM 서비스는 기본적으로 사용자 정의 정책을 1,500개 허용한다.
    -   GCP IAM은 프로젝트 내 사용자 정의 역할에 제한(300개)이 있다. 일부 제한은 유연하므로 클라우드 제공업체에 확장을 요청할 수 있다.

### 4\. 권한 유무에 따른 실무 예시 (AWS S3)

-   **권한이 없을 때 조회 작업 수행 시 예외 발생**:

```
# 예제 7-16: 권한 없이 조회 작업을 수행할 때 발생한 예외
$ aws s3 ls s3://dedp-visits-301JQN/
An error occurred (AccessDenied) when calling the ListObjectsV2 operation: Access Denied
```

-   **해결을 위해 ID 기반 접근 방식으로 S3 읽기 작업 범위 지정 역활 생성:**

```
/* 예제 7-17: S3 버킷을 조회하는 데 필요한 권한 */
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "VisitsS3Reader",
    "Effect": "Allow",
    "Action": ["s3:Get*", "s3:List*"],
    "Resource": ["arn:aws:s3:::dedp-visits-301JQN/*", "arn:aws:s3:::dedp-visits-301JQN"]
  }]
}
```

-   **대안으로 자원 기반 접근 방식을 사용하여 S3 버킷 권한 부여:**

```
# 예제 7-18: S3 버킷에서 객체를 읽을 수 있도록 AWS 역할에 권한을 부여하는 명령
$ aws s3api put-bucket-policy --bucket dedp-visits-301JQN --policy file://policy.json

# policy.json 내용
{
  "Statement": [{
    "Effect": "Allow",
    "Principal": {"AWS": "arn:aws:iam::123456789012:user/visits-s3-reader"},
    "Action": ["s3:Get*", "s3:List*"],
    "Resource": "arn:aws:s3:::dedp-visits-301JQN/*"
  }]
}
```

## 패턴 #45: 암호화기 

### 1\. 개요 및 문제 정의

-   **개요**: 논리적 수준에서, 즉 데이터베이스나 클라우드 서비스 범위에서 데이터 접근을 제어하는 것은 완전히 안전한 시스템을 구축하기에 충분하다고 생각하기 쉽지만 착각일 뿐이다. 누락된 부분은 데이터 자체를 보안하는 것, 다시 말해 예상치 못한 사용으로부터 보호하는 것이다. 클라우드에서 인프라를 실행하더라도 데이터는 여전히 어딘가에 물리적으로 저장되어 있으며 권한이 없는 사람들이 이를 읽으려고 할 수 있다. 접근 제어가 손상되더라도 데이터를 사용할 수 없도록 해야 한다.
-   **문제**: 테이블과 클라우드 자원 모두에 대해 세밀한 접근 정책을 구현한 후, 저장된 데이터와 전송 중인 데이터의 보안을 강화하는 임무를 맡게 되었다. 이해관계자들은 권한이 없는 사람이 스트리밍 브로커와 작업 간에 전송되는 데이터를 가로채거나 서버에서 물리적으로 데이터를 도난당할 수 있다고 우려하고 있다.

### 2\. 해결책 및 구현 방식

데이터 침입 위험을 줄이는 한 가지 방법은 **암호화기(Encryptor)** 패턴을 사용하는 것이다. 두 가지 보호 수준이 필요하므로 패턴에는 두 가지 구현이 존재한다.

#### \[1\] 저장된 데이터(Data-at-rest)를 위한 암호화

저장된 데이터를 암호화하는 방식은 크게 클라이언트 측 접근 방식과 서버 측 접근 방식으로 나뉘며, 두 방식의 차이점은 **암호화 키 관리**로 귀결된다.

-   **클라이언트 측 암호화**: 데이터 프로듀서가 저장소로 보내기 전에 데이터를 암호화할 책임이 있으며, 당연히 암호화 키 관리에 대한 책임도 함께 갖는다.
-   **서버 측 암호화**: 모든 암호화와 복호화 작업이 서버 측에서 수행된다. 컨슈머나 프로듀서는 요청만 발행하고, 서버가 암호화 키 관리를 포함한 나머지를 수행한다. 퍼블릭 클라우드 제공업체에서 널리 지원되며 각각 자체 암호화 키 스토어를 가지고 있다. (AWS와 GCP에서는 키 관리 서비스(KMS)를 사용하고, 애저에서는 **Key Vault** 서비스를 사용한다.)

#### \[2\] 암호화 저장 서비스를 사용한 저장 데이터 서버 측 암호화 워크플로 (그림 7-5)

클라이언트와 암호화된 데이터 스토어 간의 상호 작용은 네 가지 주요 단계를 거치며, 클라우드 제공업체에 의해 완전히 추상화된다.

1.  요청은 먼저 암호화된 데이터 스토어에 도달한다. 하지만 바로 반환되지 않는다.
2.  데이터를 반환하는 대신 데이터 스토어는 암호화 키를 식별하고, 암호화 키 스토어에 복호화 키를 요청한다. 만약 클라이언트에게 데이터 복호화 권한이 없으면 요청이 실패하고, 권한이 있다면 계속 진행된다.
3.  다음으로 데이터 스토어는 검색된 복호화 키로 데이터를 복호화한다.
4.  마지막으로 복호화된 레코드가 클라이언트에게 다시 전송된다.

-   **사용자의 유일한 책임**: 데이터 스토어에 대한 적절한 암호화 전략을 구성하고, 모든 읽기 및 쓰기 클라이언트에 대해 데이터 스토어와 암호화 키 스토어에 대한 접근을 관리하는 것이다.

#### \[3\] 전송 중인 데이터(Data-in-transit)를 위한 암호화

-   저장 데이터만이 유일한 암호화 계층은 아니다. 또 다른 계층은 클라이언트와 데이터 스토어가 네트워크를 통해 데이터를 교환하는 곳인 전송 중인 데이터이다.
-   **구현**: 클라우드에서의 구현은 클라이언트 측에서 **SDK 수준의 보안 통신을 활성화**하고 서비스에서 필요한 프로토콜 버전을 구성하는 것으로 제한되므로 비교적 쉽다. 이렇게 하면 교환되는 데이터가 암호화된 상태로 유지된다.

### 3\. 패턴의 결과 및 제한 사항

암호화는 물리적 데이터 스토어 수준에 적용되는 데이터 접근 보안 전략이다. 다른 보안 보호와 마찬가지로 무료로 얻을 수는 없다.

-   **암호화/복호화 오버헤드**: 가장 먼저 떠오르는 결과는 **CPU 오버헤드**이다. 데이터는 평문으로 저장되지 않고 어떤 식으로든 변경된 읽을 수 없는 버전으로 저장된다. 암호화와 복호화 작업 없이는 사용할 수 없기 때문에, 이러한 이유로 각 쓰기 및 읽기 요청은 CPU에 추가 압력을 가한다.
-   **데이터 손실 위험**: 암호화기 패턴이 권한이 없는 접근으로부터 저장 데이터를 보호하는 경향이 있지만, 부수 효과로 권한이 있는 사용자의 접근도 차단할 수 있다. 이는 암호화 키를 분실하거나 단순히 접근 권한을 잃어버린 경우에만 발생할 수 있다.
    -   _완화 방안_: 이 문제를 완화하기 위해 클라우드 제공업체는 종종 암호화 스토어에서 **소프트 삭제**를 구현한다. 즉, 보내는 모든 삭제 요청이 즉시 효력을 발생시키지 않는다는 의미이다. 대신 실수로 삭제한 경우 키를 복원할 수 있는 유예 기간의 혜택을 받는다.
-   **프로토콜 갱신**: 전송 중 암호화는 저장 시 암호화보다 설정하기가 훨씬 쉽다. 하지만 여전히 최신 상태로 유지되어야 하는 시스템의 추가 구성 요소이다. HTTPS에서 가능한 암호화 채널인 **전송 계층 보안(TLS) 프로토콜**이 그런 경우이다. 수년에 걸쳐 다양한 보안 취약점이 발견되었고 그 결과 TLS 1.0과 1.1 버전은 더 이상 권장되지 않아 사용이 중단되었다. (자세한 내용은 RFC 8996에서 확인 가능). 따라서 과거 버전을 사용하는 모든 서비스는 업그레이드되어야 한다. 이 작업은 유지 관리에 고충을 더하는 것 같지만, 클라우드 서비스에서는 매우 자주 단순화되어 서비스 수준에서 새로운 암호화 프로토콜 버전을 업그레이드하는 것으로 축소된다.

### 4\. 실무 구현 예시 (테라폼 코드)

-   **AWS S3 객체 스토어 저장 시 암호화 구성**: 테라폼 코드를 사용하여 실제로 작동하는 방식을 정의한다. AWS KMS 서비스에서 암호화 키를 정의하고 AWS 람다 사용자에게 복호화 및 암호화 권한을 부여한다.

```
# 예제 7-19: AWS KMS 암호화를 위한 테라폼 정의
module "kms" {
  source = "terraform-aws-modules/kms/aws"
  key_usage = "ENCRYPT_DECRYPT"
  deletion_window_in_days = 14
  aliases = ["visits-bucket-encryption-key"]
  grants = {
    lambda_doc_convert = {
      grantee_principal = aws_iam_role.iam_key_reader.arn
      operations = ["Encrypt", "Decrypt", "GenerateDataKey"]
    }
  }
}
```

-   핵심 매개변수 설명: AWS KMS 키 선언에서 신비롭지만 필수적인 유일한 매개변수는 deletion\_window\_in\_days = 14이다. 이는 데이터 손실 단점에서 언급했듯이, 잘못된 제거의 경우 암호화 키 복원 윈도(유예 기간)를 보장하기 위해 존재한다. 또한 이 선언은 다른 서비스에 일부 권한을 할당하며, 예제에서의 권한은 나중에 컴퓨팅이나 쿼리 서비스와 연결할 수 있는 IAM 역할에 적용된다.
