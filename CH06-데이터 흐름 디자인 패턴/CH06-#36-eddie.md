## 6.3 팬아웃(Fan-Out)

- 패턴#37 병렬 분할(Parallel Split)
- 직전 서브챕터(팬인)는 여러 브랜치를 하나로 합쳤다.
- 팬아웃은 반대로, 한 태스크 결과를 여러 브랜치로 **가른다**.

---

## 패턴#36 병렬 분할(Parallel Split)

### (1)문제상황

- 결론: 마이그레이션 기간에 같은 결과를 두 곳에 동시에 써야 하는 상황이다.
- 기존 방식과 실패 장면:
    - 레거시 데이터 처리 프레임워크가 C#으로 짜여 있다. 아는 사람이 다 퇴사했고 문서도 없다.
    - 리버스 엔지니어링으로 로직을 파악해, 모던 Python 라이브러리로 재작성 중이다.
- 개선 욕구:
    - 리버스 엔지니어링이 완벽하지 않을 수 있다.
    - 소비자가 신규 결과로 갈아탈 때까지 구 파이프라인도 계속 돌리고 싶다.
- 갈라지며 생긴 새 문제 (이 패턴이 푸는 지점):
    - 같은 처리 결과를 구/신 두 저장소에 동시에 써야 한다.
    - 부모 태스크(공통 처리) 하나에서 자식 두 개(구 저장소 쓰기, 신 저장소 쓰기)가 갈라진다.
    - 이 "하나 → 여럿" 분기가 병렬 분할이다.

### (2)솔루션

- 주요컨셉: 공통 연산은 부모가 1회만 수행하고, 그 결과를 여러 자식이 병렬로 나눠 쓴다.
- 오케스트레이션 계층:
    - 부모 → 자식 여럿 연결. DSL/함수로 간단히 표현.
- 처리 계층 3대 주의점:
    - 공통 연산 중복 방지: 분할이 원천 재읽기를 유발하면 안 됨. 중간 데이터셋을 materialize(SQL 임시 테이블 / Spark `.persist()`)해 1회만 계산.
    - 브랜치 간섭 방지: 공유·전역 변수는 읽기 전용이거나 모든 쓰기에 호환되게.
    - 자원 분리: 실행 시간 민감하면 브랜치별 전용 컴퓨트 할당 또는 오토스케일.

### (3)결과

- 결론: 분기 대가로 gotcha 2가지.
- 실행 차단(Blocked execution):
    - 배경: 시간 의존 파이프라인은 이전 실행 성공해야 다음이 돈다.
    - 문제: 병렬 분할의 트리거 조건은 가장 느린 브랜치에 묶인다. 한 브랜치 실패 시 후속 실행 전체가 멈춤.
    - 대응: 느린 브랜치를 별도 파이프라인으로 분리(독립 시퀀서의 데이터/태스크 기반 의존 활용).
- 하드웨어 불일치:
    - 배경: 한 부모가 두 자식용 중간 데이터셋을 만든다.
    - 문제: 자식들이 요구 자원이 다르면(하나는 CPU 중심, 하나는 메모리 중심) 단일 인프라로 못 맞춤.
    - 대응: 중간 데이터셋 생성과 병렬 잡을 분리해, 각 잡을 전용 하드웨어에서 실행.

### (4)예시

- 결론: 오케스트레이션은 부모→자식 여럿 연결, 처리 계층은 중간 결과 materialize가 핵심.
- 예시) 병렬 분할 (Example 6-12, Airflow DSL)

```
# 공통 부모 → 두 자식으로 분기
generate_dataset >> [write_to_legacy_store, write_to_new_store]
```

- 설명: 부모 `generate_dataset` 하나가 두 자식으로 갈라진다.
- ★ 핵심: `[write_to_legacy_store, write_to_new_store]` 리스트 표기. 부모 완료 후 두 자식이 병렬 실행.
- 처리 계층 핵심 (Spark):

```
df = spark.read.parquet("s3://.../cleaned/")   # 공통 원천 1회 읽기
df.persist()                                    # ★ 핵심: 중간 결과 캐시
df.write.parquet("s3://.../legacy/")            # 자식1
df.write.parquet("s3://.../new/")               # 자식2
```

- 설명: `df`를 두 번 쓰지만, `persist()` 덕에 원천을 다시 읽지 않는다.
- ★ 핵심: `.persist()`. 없으면 각 write가 원천을 재읽기·재계산해 공통 연산이 2배로 돈다.
- 엔지니어 독백:
    
    > 신입한테. 병렬 분할에서 `persist()` 빼먹으면 로그상 잘 도는데 원천을 브랜치 수만큼 다시 읽는다. 소스가 큰 파케이면 이게 그냥 2배, 3배 요금이다. 나는 한 DataFrame을 두 번 이상 write하면 무조건 persist 걸었나부터 본다.
    

### (5)최신트렌드

- 결론: 수동 분기·캐시를 자동 최적화·선언적 팬아웃으로 대체하는 흐름.
- Spark 캐시 자동화 (AQE + reuse exchange):
    - 배경: `persist()` 위치·해제를 수동 관리해 실수가 잦았다.
    - 해결: 동일 연산 재사용(reuse exchange)을 옵티마이저가 자동 감지. 중복 계산 축소.
- dbt (팬아웃 모델 의존):
    - 배경: 한 결과를 여러 다운스트림에 수동 연결하면 의존이 코드에 안 드러났다.
    - 해결: 하나의 모델을 여러 모델이 `ref()`로 참조 → 팬아웃 의존을 선언적으로 표현.
- 실무 선호:
    - Spark 파이프라인 → AQE 켜고 persist는 명시적으로 병행. 자동+수동 이중 안전장치.
    - SQL 중심 → dbt로 팬아웃 의존 관리.
- 엔지니어 독백:
    
    > 신입한테. 마이그레이션용 이중 쓰기는 "언제 끊을지"를 처음부터 정해라. 구 저장소 쓰기를 한번 켜두면 아무도 안 끄고 1년씩 돈다. 나는 이중 쓰기 브랜치에 항상 종료 날짜를 티켓으로 박아둔다.
