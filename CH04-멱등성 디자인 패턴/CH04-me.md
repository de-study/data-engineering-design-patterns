# 4. 멱등성 디자인 패턴

멱등성의 목적

- 데이터 처리 잡을 여러 번 실행해도 매번 중복된 데이터가 생기지 않거나
- 설사 생기더라도 명확히 식별 가능

# 4.1 덮어쓰기

## 패턴 #16: 빠른 메타데이터 정리기

메타데이터 작업은 데이터 파일과 상호 작용할 필요가 없어서 빠르지만 데이터 파일을 설명하는 더 작은 계층에서 작동 → 논리적 수준에서 작동한다고 할 수 있음

### 4.1.1 문제

- delete/insert 구현시 시간이 지날수록 데이터셋이 커지고 delete 태스크 성능이 나빠짐

DELETE 문제점

- DELETE는 테이블 스키마는 그대로 두고 안에 있는 데이터를 한 줄씩 지우는 명령어
- 느리고 무거움: 데이터를 지울 때마다 delete 기록(로그)를 디스크에 한 줄씩 상세히 기록
- 디스크 공간이 그대로 남음: 데이터는 지워지지만 그 데이터가 차지하고 있던 디스크 공간(컴퓨터 메모리 공간)은 그대로 남아있음
- 멱등성 관점의 위험성: 데이터를 지우는데도 오래걸리고 지우는 도중 문제가 생기면 일부만 지워진 모호한 상태가 됌

### 4.1.2 해결책

- drop과 truncate는 모든 레코드를 제거한다는 점에서 같지만, truncate는 테이블을 스캔하지 않는 다는 점에서 다르고, 그렇기때문에 “메타데이터 작업”으로 분류

1. DROP TABLE

- 데이터뿐만 아니라 테이블의 이름, 스키마, 인덱스까지 데이터베이스에서 삭제
- 삭제하는 동안 락이 걸리기 때문(원자적 처리)에 사용자가 접근할 수 있는 view를 제공해야 될 수 있음

2. TRUNCATE TABLE

- 데이터만 삭제후 재적재하는 로직
- 데이터의 상세 로그를 남기지 않고 데이터가 저장된 디스크 공간의 포인터(위치 정보) 자체를 해제
- 디스크 공간도 즉시 반환

빠른 메타데이터 정리기 패턴은 멱등성 달성을 위해 아래 두가지에 의존

- 데이터셋 파티셔닝
    - 파티션 크기만큼 테이블을 정리하기 위해 메타데이터 작업을 적용할 수 있는 단위도 동시에 정의 (백필링과 관련)
- 데이터 오케스트레이션
    - 예를 들어 월요일이면 새로운 파티션 생성하여 데이터 적재, 월요일이 아니라면 기존 파티션에 데이터 적재
    - 월요일이라면 안전하게 재작업 할 수 있는 깨끗한 상태로 만드는 작업 필요 - drop table, truncate table
    - 주별 테이블을 새로 만들거나 비웠다면 사용자들이 데이터를 조회할 때. ㅏ용하는 하나의 공통 뷰를 최신 상태로 갱신해줘야 함.

### 4.1.3 결과

#### 1. 세분화와 백필링 관계

- 백필링 세분화이기도 한 멱등성 세분화를 정의 → 즉 파이프라인을 리플레이하려면 파티션된 테이블을 생성하는 태스크부터 리플레이해야 함
- 촘촘한 백필: 데이터 제공업체, 사용자 고객이 하나일 때 메타데이터 작업이 항상 전체 테이블을 대상으로 하므로 빠른 메타데이터 정리기 패턴이 유용하지 않음

#### 2. 메타데이터 제한

- 무한 생성이 불가능
- GCP 빅쿼리, AWS Redshift같은 데이터 웨어하우스는 각각 4000개 파티션과 200,000개 테이블로 제한
- 무한 생성 불가능 문제 해결을 위해 멱등성 테이블을 불변의 것으로 변환하는 동결 단계 추가 가능
    - 예를 들어 주별 테이블을 동결 기간 후에 변경 사항이 없다면 월간 또는 연간 테이블로 변경
- 빠른 메타데이터 정리기 패턴은 메타데이터 작업을 지원하는 데이터베이스에서만 작동 - 데이터 웨어하우스, 레이크하우스, 관계형 데이터베이스
    - 객체 스토어는 데이터 덮어쓰기 패턴에 의존해야 함

#### 3. 데이터 노출 계층

- 하나의 엔트리에서 데이터에 접근하기를 선호한다면 통합 뷰 등으로 노출하는 논리적 구조를 해결책으로 사용 가능

#### 4. 스키마 진화

- 빠른 메타데이터 정리기 패턴에서는 기존 테이블의 스키마 갱신을 위해 선택적 필드를 새로 추가하면 데이터를 재처리하기 때문에 효과적이지 않음
- 기존 데이터를 재처리하지 않고 스키마를 진화 시키는 핵심은 과거 데이터 파일의 물리적 구조를 절대 건드리지 않는 것
    - 스키마 레지스트리: 데이터와 스키마를 분리해 버전별로 관리하고 애플리케이션 단에서 호환
    - 뷰 추상화: 구 데이터 테이블과 신 데이터 테이블을 분리하고 뷰로 묶어서 제공
    - 최신 포맷(Iceberg 등) 활용: 파일 내부 ID 매핑 기능을 통해 메타데이터 레이어만 갱신하고 기존 파일은 유지
    - 하이브리드 스키마: 가변적인 필드는 JSON 컬럼 하나에 몰아넣어 테이블 구조 변경 자체를 방지

## 패턴 #17: 데이터 덮어쓰기

### 4.1.5 문제

- 객체 스토어의 이벤트 시간으로 파티션된 위치에 저장된 데이터셋의 백필링

### 4.1.6 해결책

#### 1. 데이터 처리 프레임워크

- wrtier 설정할때 옵션 설정 가능
    - 스파크: savemode
    - 플링크: writemode
    - 델타 레이크: replaceWhere - 지정된 필터링 조건에 맞는 데이터셋의 일부만 덮어씀

#### 2. SQL

- DELETE FROM + INSERT INTO
- INSERT OVERWRITE
- 구글 빅쿼리: LOAD DATA OVERWRITE

### 3. 타임트레블

- 타임 트레블 기능을 지원하는 경우 데이터셋을 과거 버전 중 하나로 복원 가능
- 설정된 보존 기간 이후나 사용되지 않는 공간을 되찾기 위해 수행하는 베큠 작업을 실행한 후에만 삭제
- GCP 구글 빅쿼리, 스노우 플레이크 - 데이터 스토어가 도중에 데이터를 삭제하지 않음

### 4.1.7 결과

#### 1. 데이터 오버헤드

- 데이터 파티셔닝이 선행되지 않으면 성능 저하

#### 2. 배큠의 필요

- A DELETE 연산은 데이터를 즉시 디스크에서 삭제하지 않을 수 있음
- 죽은 레코드가 점유한 공간 회수를 위해 배큠 수행 필요
