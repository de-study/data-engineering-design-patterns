데이터 엔지니어링을 하다 보면 가장 까다로운 작업 중 하나가 바로 '이미 존재하는 데이터의 변경 사항을 어떻게 반영할 것인가?'이다.  
  
데이터셋을 완전히 삭제하고 새로 교체하는 '덮어쓰기(Overwrite)' 방식은 멱등성(Idempotency)을 보장하는 가장 단순하고 확실한 방법이다. 하지만 데이터 규모가 커지거나, 데이터 제공자가 변경·수정된 일부 데이터만 포함하는 '증분형 데이터셋'을 제공할 때는 전체 덮어쓰기 방식으로 쉽게 관리할 수 없다.

### 1\. 병합(Merge, Upsert) 패턴이란?

병합 패턴은 데이터셋의 정체성이 정적이고 데이터셋이 갱신이나 삽입만을 지원할 때, 기존 데이터셋과 변경 사항을 결합하는 최선의 방법이다. 흔히 MERGE(속칭 UPSERT) 명령어로 표현된다.

**1.1 구현을 위한 단계적 프로세스**

1.  **고유 속성 정의**: 데이터셋 전체에서 고유성을 보장하기 위한 속성을 정의하는 단계이다. 사용자 ID 같은 단일 속성이나, 웹사이트 방문 이벤트의 경우 방문 ID + 방문 시간 같은 여러 속성의 조합을 사용할 수 있다.
2.  **처리 계층 프레임워크 활용**: 테이블 파일 형식, 데이터 웨어하우스 등 현재 널리 사용되는 대부분의 해결책은 MERGE 명령을 지원한다.
3.  **시나리오별 동작 정의**: 올바른 실행 방법을 찾았다면 다음 세 가지 시나리오에서 취할 동작을 정의해야 한다.
    -   **삽입(Insert)**: 새 데이터셋의 엔트리가 현재 데이터셋에 존재하지 않을 때 새 레코드를 추가한다.
    -   **갱신(Update)**: 두 데이터셋이 주어진 레코드를 저장하고 있지만, 새 데이터셋이 그 레코드의 갱신된 버전을 제공할 때 수정 작업을 수행한다.
    -   **삭제(Delete)**: 아래에서 상세히 다룬다.

**1.2 병합 패턴에서 가장 까다로운 '삭제(Delete)' 처리하기**

병합 패턴은 소스 데이터셋에서 레코드가 누락되어도 타깃 데이터셋에 아무런 일이 일어나지 않기 때문에 삭제를 지원하지 않는 경우가 많아 가장 까다롭다. 이 때문에 데이터에 물리 삭제나 논리 삭제를 적용하려면 **'논리 삭제(주어진 레코드를 삭제된 것으로 표시하는 속성을 가진 갱신)'** 형태로 표현해야 변화를 감지할 수 있다.

해결책: is\_deleted 플래그를 활용한 논리 삭제 구현  
Apache Spark 등에서의 논리 삭제 구현 예시는 다음과 같다.

```
MERGE INTO dedp.devices_output AS target
USING dedp.devices_input AS input
ON target.type = input.type AND target.version = input.version
WHEN MATCHED AND input.is_deleted = true THEN
  DELETE
WHEN MATCHED AND input.is_deleted = false THEN
  UPDATE SET full_name = input.full_name
WHEN NOT MATCHED AND input.is_deleted = false THEN
  INSERT (full_name, version, type) VALUES (input.full_name, input.version, input.type)
```

> 주의 사항: INSERT 문에서 input.is\_deleted = false 플래그를 조건으로 사용하는 것이 매우 중요하다. 그렇지 않으면 병합 패턴의 첫 실행 동안 과거에 이미 제거된 레코드가 다시 삽입될 수 있으며, 이로 인해 결과적으로는 데이터가 영원히 제거되지 못하는 현상이 발생할 수 있다.

**1.3 병합 패턴의 트레이드오프와 '백필(Backfill)'의 함정**

병합 패턴은 단순해 보이지만 데이터셋의 특성(증분 또는 전체)에서 비롯된 함정과 트레이드오프가 존재한다.

-   **고유함(Identity)**: 가장 중요한 1순위 요구사항이다. 각 레코드를 안전하게 식별할 수 있는 불변의 속성을 정의해야 하며, 그렇지 않으면 백필링의 경우 레코드를 갱신하는 대신 새 레코드를 삽입해 일관성 없는 중복이 발생할 수 있다.
-   **입출력(I/O)**: 병합 패턴은 메타데이터 정리 패턴과 달리 데이터 블록 수준에서 직접 작동하며 매우 컴퓨팅 집약적이다. 다만 최신 데이터베이스와 테이블 파일 형식은 메타데이터 계층에서 영향받는 레코드를 먼저 검색해 관련 없는 파일 처리를 건너뛰도록 최적화한다.
-   **백필링을 사용한 증분 데이터셋의 일관성 문제**: 이 패턴의 치명적인 단점은 백필 컨텍스트에서 일관성이 부족해진다는 점이다.

**데이터가 꼬이는 백필 시나리오 분석**

정상적인 증분 데이터 로딩 (U=갱신, D=삭제)

-   **07:00** | 새 레코드: A | 출력 테이블 레코드: A
-   **08:00** | 새 레코드: A-U, B | 출력 테이블 레코드: A-U, B
-   **09:00** | 새 레코드: B-D, C | 출력 테이블 레코드: A-U, C (B는 논리 삭제됨)
-   **10:00** | 새 레코드: M, N, O | 출력 테이블 레코드: A-U, C, M, N, O

만약 어떠한 이유로 **08:00부터 파이프라인을 리플레이(백필)** 해야 한다고 가정해본다. 이미 출력 데이터셋은 10:00 기준의 최신 버전(A-U, C, M, N, O)까지 가 있는 상태이다.

병합 패턴으로 다시 백필을 수행한 후의 결과

-   **08:00 백필 실행** | 새 레코드: A-U, B | 현재 레코드: A-U, C, M, N, O | 출력 테이블 레코드: A-U, B, C, M, N, O

> **일관성 붕괴**: 원래대로라면 09:00에 삭제되었어야 할 레코드 **B가 테이블 내에 다시 살아나게 된다**. 백필 과정에서 파이프라인이 09:00에 도달하여 다시 삭제하기 전까지, 컨슈머는 실제 데이터와 동일하지 않은 상태의 출력 레코드를 보게 되며 지연 기간을 거쳐 서서히 정상으로 돌아온다.

**1.4 결론 및 보완책**

이 문제를 완화하려면 테이블을 첫 번째 리플레이된 실행으로 환원하는 복원 메커니즘을 파이프라인 외부 또는 내부에 구현해야 한다. 데이터베이스나 테이블 파일 형식이 지원하는 타임 트래블(Time Travel) 기능을 활용하면 이러한 버전 관리 기능을 통해 비교적 쉽게 구현할 수 있다.

병합(Merge) 패턴의 요약

기본 병합 패턴은 타깃 테이블의 고유 키(Primary Key)를 기준으로 새 데이터가 들어오면 INSERT를, 기존 데이터가 존재하면 최신 값으로 UPDATE를 수행하는 방식이다.

-   **삭제(Delete)의 한계**: 소스 데이터셋에서 레코드가 누락되어도 타깃 데이터셋에는 아무런 일이 일어나지 않는다. 따라서 완벽한 동기화를 위해서는 데이터 소스 단에서 is\_deleted = true와 같은 플래그를 넘겨주는 **'논리 삭제'** 방식을 연계해야 한다.
-   **백필 시의 일관성 붕괴**: 이미 10:00 시점의 최신 데이터가 반영된 출력 테이블에, 시스템 장애 등으로 인해 08:00 시점의 과거 증분 데이터를 다시 밀어 넣으면(리플레이), 중간에 삭제되었어야 할 레코드가 테이블 내에 다시 살아나는 등 데이터셋의 일관성이 완전히 무너진다.

  
만약 상태 비저장 병합 패턴이 지닌 백필 시의 일관성 부족 문제를 극복하고 복원 요구사항을 채우고자 한다면, 이 패턴을 확장한 '패턴 #19: 상태 저장 병합 패턴'을 시도해보는 것이 좋다.

데이터 파이프라인을 운영할 때 가장 까다로운 도전 과제 중 하나는 변경·수정된 일부 데이터만 포함하는 '증분형 데이터셋'을 기존 데이터셋에 어떻게 안전하게 반영할 것인가이다.  
  
앞서 살펴본 기본 병합(Merge, 속칭 UPSERT) 패턴은 매우 직관적이지만, 과거 특정 시점부터 파이프라인을 재처리해야 하는 백필(Backfill) 컨텍스트에서 데이터 일관성이 무너지는 치명적인 단점이 존재한다.

### 2\. 상태 저장 병합기(Stateful Merger) 패턴

기본 병합 패턴의 백필 시 일관성 부족 문제를 해결하기 위해, 추가적인 '상태 테이블(State Table)'을 두어 데이터 복원 기능을 파이프라인 워크플로 내에 내재화하는 방식이 바로 **상태 저장 병합기 패턴**이다.

**2.1 상태 저장 병합기의 기본 워크플로**

파이프라인이 실행될 때 단순히 MERGE 문만 실행하는 것이 아니라, 앞뒤로 '복원'과 '상태 기록' 태스크가 추가된다.

1.  **필요한 경우 테이블 복원 (시작 단계)**: 현재 실행이 백필 모드인지 감지하고, 필요한 경우 메인 테이블을 과거의 유효한 버전으로 롤백한다.
2.  **병합 문 실행 (중간 단계)**: 복원된 테이블 위에 새 증분 데이터를 병합(MERGE)한다.
3.  **상태 테이블 갱신 (마지막 단계)**: 작업이 완료되면 새로 생성된 테이블의 버전 번호와 실행 시간을 상태 테이블에 매핑하여 기록한다.

**2.2 백필링 감지 및 복원 알고리즘의 작동 원리**

데이터 오케스트레이터(예: Apache Airflow)의 컨텍스트 메타데이터를 분석하거나, 파이프라인 내부에 **상태 테이블**을 조회하는 고수준 로직을 구현하여 백필 여부를 판별한다.

백필 판별 및 복원 단계 (2단계 프로세스)

1.  **이전 버전 가져오기**: 이전 파이프라인 실행에 의해 생성된 테이블 버전을 조회한다. 만약 기록된 버전이 없다면 완전히 처음 실행하거나 데이터가 없는 상태를 의미하므로, TRUNCATE TABLE 명령어로 데이터를 비우고 바로 병합 작업으로 이동한다.
2.  **버전 비교 분석**:
    -   **두 버전이 같은 경우 (정상 모드)**: 오케스트레이터의 현재 실행 시간이 최신 실행 시간과 동일하다면 파이프라인이 정상 순차 모드로 동작하고 있음을 의미하므로 복원 없이 바로 병합을 수행한다.
    -   **두 버전이 다른 경우 (백필 모드)**: 현재 실행하려는 시간의 과거 기록 버전과 시스템의 최신 버전 간에 차이가 감지된다면, 이는 과거 시점을 재처리하는 백필 시나리오에 진입했음을 의미한다. 따라서 병합을 적용하기 전에 **테이블을 이전 버전으로 복원(Rollback)** 해야 한다.

컴팩션(Compaction) 작업이 존재할 때의 예외 케이스 처리

데이터 레이크 환경에서는 작은 파일들을 큰 파일로 병합하는 컴팩션(Compaction) 작업이 수시로 일어난다. 컴팩션은 데이터를 덮어쓰지 않는 메타데이터 작업(no-data action)이지만, 델타 레이크 시스템상 테이블의 버전 번호는 상승하게 된다.

-   **문제 상황**: 상태 테이블에는 파이프라인 실행 시간 기준의 버전만 기록되므로, 그 사이에 컴팩션으로 인해 올라간 버전 번호는 누락될 수 있다.
-   **해결책**: 단순히 상태 테이블에 기록된 과거 버전을 그대로 조회하는 대신, "현재 실행 시간에 해당하는 버전에서 1을 차감하는 방식(version\_for\_current\_execution\_time - 1)"으로 로직을 변경한다. 이렇게 하면 파이프라인 사이에 컴팩션 같은 데이터 없는 작업이 수행되었더라도 정확히 직전의 유효한 데이터 커밋 버전으로 안정적인 복원이 가능하다.

**2.3 PySpark & Delta Lake 구현 코드 분석**  
이 패턴이 스파크와 델타 레이크 환경에서 어떻게 코드로 구현되는지 세부 태스크별로 살펴본다.  
  
① 상태 테이블 정의  
잡(Job)의 실행 시간과 그로 인해 생성된 델타 테이블의 버전을 저장할 메타 테이블을 생성한다.

```
CREATE TABLE IF NOT EXISTS `default`.`versions` (
  execution_time STRING NOT NULL,
  delta_table_version INT NOT NULL
)
```

② 현재 및 과거 버전 조회 (복원 판단)  
델타 레이크의 히스토리에서 마지막 MERGE 버전을 조회하고, 상태 테이블에서 이전 실행 시점의 버전을 가져와 비교한다.

```
# 메인 테이블의 최신 MERGE 버전 확인
last_merge_version = (spark.sql('DESCRIBE HISTORY default.devices')
                      .filter("operation = 'MERGE'")
                      .selectExpr('MAX(version) AS last_version').collect()[0].last_version)

# 상태 테이블에서 과거 실행 시간에 매핑된 버전 확인
maybe_previous_job_version = spark.sql(f"""
    SELECT delta_table_version FROM versions 
    WHERE execution_time = "{previous_execution_time}"
""").collect()
```

③ 데이터 복원 및 테이블 환원 작업  
조회 결과 과거 버전 정보가 없다면 테이블을 비우고(TRUNCATE), 백필 모드임이 확인되면 restoreToVersion API를 호출해 해당 버전 직전 상태로 타임 트래블 복원을 수행한다.

```
if not maybe_previous_job_version:
    # 최초 실행 혹은 데이터가 없는 경우 테이블 초기화
    spark.sql('TRUNCATE TABLE default.devices')
else:
    previous_job_version = maybe_previous_job_version[0].delta_table_version
    # 과거 버전이 최신 MERGE 버전보다 낮다면 백필 모드로 판단하여 복원 수행
    if previous_job_version < last_merge_version:
        current_run_version = (spark.sql(f"""
            SELECT delta_table_version FROM versions 
            WHERE execution_time = "{currently_processed_version}"
        """).collect()[0].delta_table_version)
        
        # 컴팩션 등의 영향을 고려하여 1을 차감한 버전으로 안전하게 복원
        version_to_restore = current_run_version - 1
        from delta.tables import DeltaTable
        deltaTable = DeltaTable.forName(spark, 'devices')
        deltaTable.restoreToVersion(version_to_restore)
```

④ 병합 후 상태 테이블 업데이트 (마지막 단계)  
MERGE 작업이 성공적으로 완료되면, 방금 생성된 최신 테이블 버전 번호를 상태 테이블에 기록하여 다음 실행 시 복원 지점으로 사용할 수 있도록 저장한다.

```
# MERGE 완료 후 새롭게 생성된 최신 버전 번호 추출
last_version = (spark.sql('DESCRIBE HISTORY default.devices')
                .selectExpr('MAX(version) AS last_version').collect()[0].last_version)

# 새 버전 정보를 DataFrame으로 생성
new_version_df = spark.createDataFrame([
    Row(execution_time=current_execution_time, delta_table_version=last_version)
])

# 상태 테이블(versions)에 새 버전 정보를 병합(Upsert)하여 기록 갱신
(DeltaTable.forName(spark_session, 'versions').alias('old_versions')
 .merge(new_version_df.alias('new_versions'), 'old_versions.execution_time = new_versions.execution_time')
 .whenMatchedUpdateAll()
 .whenNotMatchedInsertAll()
 .execute())
```

**2.4 상태 저장 병합기 패턴의 트레이드오프**  
상태 저장 병합기 패턴은 백필 시 데이터 일관성이 깨지는 문제를 완벽하게 방정해 주지만, 인프라와 비용 측면에서 다음과 같은 명확한 트레이드오프를 동반한다.  
  
버전 관리되는 데이터 스토어의 필수성: 타임 트래블 기능이 완벽히 지원되는 Delta Lake, Apache Iceberg와 같은 최신 테이블 파일 형식을 기반으로 해야만 이 패턴을 온전히 구현할 수 있다. 만약 버전 관리가 지원되지 않는 일반 RDBMS(예: PostgreSQL 등) 환경이라면 원시 데이터(Raw Data) 테이블에 실행 시간 컬럼을 두고 매번 중복을 제거하며 메인 테이블을 다시 빌드하는 복잡한 우회 로직(윈도 중복 제거 패턴 등)을 직접 구현해야 하므로 파이프라인 복잡도가 증가한다.  
  
배큠(Vacuum) 작업의 제한과 스토리지 비용: 데이터 레이크 환경은 스토리지 최적화를 위해 보존 기간이 지난 무효한 스토리지 파일을 지우는 배큠(Vacuum) 작업을 주기적으로 실행한다. 그러나 복원에 필요한 과거 특정 시점의 데이터 파일이 배큠에 의해 물리적으로 삭제된다면 타임 트래블 복원이 실패하여 백필 자체가 불가능해질 수 있다. 이를 막기 위해 데이터 보존 기간 설정을 늘려야 하며, 이는 곧 스토리지 비용의 상승으로 직결된다.  
  
따라서 비즈니스의 데이터 일관성 요구 레벨과 스토리지 비용 가용성을 종합적으로 비교 평가하여 최적의 갱신 패턴을 선택해야 한다.

### 3\. 키 기반 멱등성 패턴

트랜잭션 기능(ACID)을 완벽하게 지원하지 않는 스토리지 환경이거나, 분산 메시지 큐(Kafka 등)와 연계하여 종단 간(End-to-End) 멱등성을 달성해야 할 때는 '키 기반 멱등성 작성자 패턴'을 도입해야 한다.

**3.1 문제 정의**

네트워크 타임아웃이나 워커 노드의 일시적인 장애로 인해 이미 데이터를 성공적으로 전송했음에도 불구하고 파이프라인이 실패로 기록될 수 있다. 오케스트레이터가 이를 재시도(Retry)하거나 과거 시점을 백필할 때, 동일한 데이터가 중복으로 인입되면서 데이터 정산이 망가지는 현상이 발생한다.

**3.2 해결책: 멱등성 키(Idempotency Key)의 원자적 검증**

이 패턴은 타깃 시스템에 데이터를 밀어 넣기 전에 "이미 처리된 고유한 작업인가?"를 고유 키를 통해 판별한다.

1.  **멱등성 키 생성**: 오케스트레이터의 실행 컨텍스트 파라미터(예: 작업명 + 실행 시간 + 배치 ID)를 조합하여 고유하고 불변하는 멱등성 키(Idempotency Key)를 생성한다.
2.  **원자적 확인 및 저장(Check-and-Set)**: 타깃 데이터베이스에 데이터를 쓰기 직전, 별도의 유효성 검증 저장소(KV 스토어 또는 고유 인덱스가 지정된 메타 테이블)에 해당 멱등성 키가 존재하는지 확인한다.
    -   **키가 존재하지 않는 경우**: 처음 실행되는 유효한 작업이므로 데이터를 적재하고 키를 저장소에 등록한다.
    -   **키가 이미 존재하는 경우**: 과거에 이미 성공적으로 처리된 작업이므로, 실제 쓰기 연산을 수행하지 않고 무시(Skip)하거나 성공 메시지만 반환한다.

### 4\. 트랜잭션 기반 작성자(Transaction-Based Writer) 패턴

이 패턴은 데이터 레이크(Delta Lake, Iceberg 등)나 데이터 웨어하우스가 제공하는 트랜잭션(ACID) 경계를 파이프라인의 실행 단위와 일치시키는 방식이다.  
  
파이프라인이 데이터를 쓰는 동안 오류가 발생하면, 시스템은 처리 중이던 데이터를 단 한 줄도 남기지 않고 완전히 이전 상태로 되돌린다(Rollback). 즉, 데이터는 '전부 적재되거나(All)', '아예 적재되지 않거나(Nothing)' 둘 중 하나의 상태만 가진다.

**4.1 백필(Backfill) 컨텍스트에서의 작동 원리: 멱등성 보장**

트랜잭션 기반 작성자 패턴은 과거 데이터를 재처리할 때 발생할 수 있는 '데이터 중복' 문제를 트랜잭션 ID 또는 실행 토큰을 활용해 깔끔하게 해결한다.

**4.2 멱등성(Idempotency) 구현 로직**

1.  **고유한 트랜잭션 ID 발급**: 오케스트레이터(Airflow 등)는 파이프라인을 실행할 때마다 해당 실행 주기(예: 2026-05-28 15:00 실행 잡)에 고유한 execution\_id 또는 commit\_token을 생성한다.
2.  **트랜잭션 내에서 처리**: 데이터를 타깃 테이블에 적재할 때, 이 고유 토큰을 레코드의 메타데이터 필드에 함께 기록하며 트랜잭션을 시작한다.
3.  **조건부 커밋 또는 덮어쓰기**:
    -   만약 동일한 실행 시점(execution\_id)으로 백필이 다시 요청되면, 타깃 테이블에서 해당 토큰을 가진 기존 데이터 블록을 트랜잭션 안에서 먼저 안전하게 제거하거나 원자적으로 덮어쓴다.
    -   이를 통해 네트워크 단절이나 워커 노드 다운으로 인해 파이프라인이 중간에 실패하더라도, 재시도(Retry) 시 데이터가 중복으로 쌓이지 않고 언제나 일관된 최종 상태를 유지할 수 있다.

**4.3 PySpark & Delta Lake 기반 예제 분석**

델타 레이크는 기본적으로 파일 시스템 수준에서 ACID 트랜잭션 로그(Delta Log)를 관리하므로, 이 패턴을 가장 잘 구현할 수 있는 환경 중 하나이다.

① 트랜잭션 기반 조건부 삽입/덮어쓰기

```
# 특정 execution_id에 해당하는 과거 백필 데이터를 원자적으로 교체하기 위해
# 파티션 조건 또는 조건부 덮어쓰기(ReplaceWhere)를 트랜잭션으로 실행한다.
(changed_devices.write
 .format("delta")
 .mode("overwrite")
 .option("replaceWhere", f"execution_id = '{current_execution_id}'")
 .saveAsTable("default.devices"))
```

-   **장점**: 데이터 파이프라인의 원자성(Atomicity)이 보장되므로, 데이터가 중간까지만 들어가서 발생하는 '부분 실패' 문제를 원천 차단한다. 다운스트림 파이프라인은 항상 정합성이 검증된 커밋 데이터만 바라볼 수 있다.
-   **단점**: 분산 환경에서 대량의 데이터를 트랜잭션으로 묶을 때 메타데이터 락(Lock) 경합이 발생할 수 있으며, 지원하는 스토리지 엔진(Delta, Iceberg, BigQuery 등)에 대한 기술적 의존도가 높다.
