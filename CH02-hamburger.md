## 1\. 전체 적재 (Full Load) 패턴

**전체 적재**는 매번 소스 시스템의 데이터셋 전체를 로드하는 방식이다. 가장 단순하지만, 데이터 규모가 커지는 것을 생각해야 한다.

**주요 특징 및 해결책**

-   **사용 사례:** 데이터베이스 부트스트랩(초기 구축)이나 크기가 작은 참조 데이터셋(Reference Dataset) 생성에 유용하다.
-   **EL vs ETL:** 별도의 변환 없이 데이터 스토어 명령어로 바로 옮기면 **EL(Extract-Load)**, 이종 데이터베이스 간의 형식을 맞추기 위해 얇은 변환 계층을 두면 **ETL** 잡이 된다.
-   **데이터 일관성 문제:** 실행마다 기존 데이터를 삭제하고 새로 삽입하는(Drop-and-Insert) 과정에서 컨슈머가 데이터를 못 보거나 부분적인 데이터만 볼 위험이 있다.
-   **해결책 (단일 데이터 노출 추상화):** 트랜잭션을 지원하지 않는 경우, 두 개의 테이블을 두고 공용 뷰(View)가 가리키는 대상을 교체하는 방식으로 가시성을 관리할 수 있다.

**기술적 구현 예시**

-   **AWS S3:** aws s3 sync 명령어를 사용하여 버킷 간 데이터를 자동으로 동기화할 수 있다.
-   **Apache Spark & Delta Lake:** 대량의 데이터를 병렬 처리하며, 델타 레이크의 트랜잭션 기능을 활용해 일관성 문제를 해결할 수 있다.

## 2\. 증분 적재 (Incremental Load) 패턴

**2-1. 증분로더**

**증분 로더**는 지속적으로 증가하는 대규모 데이터셋에서 "마지막 실행 이후 추가된 부분"만 수질하는 효율적인 패턴이다.

**두 가지 핵심 구현 방식**

1.  **델타 컬럼(Delta Column) 방식:**
    -   ingestion\_time 같은 컬럼을 기준으로 마지막 수집 시점 이후의 레코드만 식별한다.
    -   주의: 이벤트 시간 지연으로 인한 데이터 누락 위험이 존재한다.
2.  **시간 분할 데이터셋(Partitioned Datasets) 방식:**
    -   데이터가 이미 시간 기반 파티션(예: 날짜별 폴더)으로 조직되어 있는 경우다.
    -   파티션 단위로 처리하기 때문에 로직이 단순해지고 최적화가 쉽다.

**고려해야 할 과제**

-   **삭제 처리:** 원본에서 레코드가 물리적으로 사라지면 증분 방식으로는 감지하기 어렵다. 이 경우 논리 삭제(Soft Delete)를 사용해 삭제된 것으로 표시(Update)하는 전략이 필요하다.
-   **백필(Backfill) 위험:** 과거 데이터를 재처리할 때 리소스가 과하게 투입될 수 있다. SQL의 BETWEEN 등을 활용해 수집 윈도우를 제한함으로써 문제를 완화할 수 있다.

**기술적 구현 예시 (Airflow & Spark)**

-   **FileSensor:** Apache Airflow에서 다음 파티션이 준비될 때까지 기다렸다가 수집 잡을 트리거하는 방식으로 파이프라인을 안정화한다.
-   **SparkKubernetesOperator:** 쿠버네티스 환경에서 스파크 잡을 실행하여 대규모 증분 데이터를 분산 처리한다.

**2-2. 변경 데이터 캡처 (Change Data Capture, CDC)**

증분 로더만으로는 부족한 **실시간성**과 **물리적 삭제 감지** 문제를 해결하기 위한 고도화된 패턴이다.

**1\. 왜 CDC가 필요할까?**

-   **지연 시간(Latency):** 증분 로더는 잡 스케줄링과 쿼리 실행 오버헤드 때문에 수 초 내의 빠른 데이터 수집이 어렵다.
-   **물리적 삭제:** 원본 DB에서 레코드가 완전히 삭제되면, 증분 로더는 무엇이 사라졌는지 알 방법이 없다.
-   **비즈니스 요구:** 데이터 변화를 30초 내에 캡처하여 중앙 토픽으로 전파해야 하는 실시간 스트리밍 요구사항이 있을 때 필수적이다.

**2. 핵심 메커니즘: 커밋 로그(Commit Log) 활용**  
DB 내부의 저수준 커밋 로그를 직접 읽어 처리하므로 매우 낮은 지연 시간을 보장한다.  
수정된 모든 레코드를 지속적으로 수집하며, 삭제(Delete) 작업까지 포함한 모든 유형의 데이터 변화를 가로챌 수 있다.  
  
**3\. CDC 도입 시 고려해야 할 과제**  
운영 복잡성: 단순 엔지니어링을 넘어 서버의 커밋 로그 활성화 등 DB 운영 팀의 밀접한 도움이 필요하다.  
데이터 범위: CDC를 시작한 시점 이후의 데이터만 캡처하므로, 전체 과거 데이터가 필요하다면 '전체 로더'와 결합해야 한다.  
페이로드(Payload): 단순 값뿐만 아니라 작업 유형(INSERT/UPDATE/DELETE), 수정 시간 등 메타데이터가 함께 들어오므로 컨슈머 측의 처리 로직이 필요하다.

주요 기술 구현 예시

1.  **Debezium (CDC):** PostgreSQL 등의 커밋 로그를 읽어 카프카 토픽으로 쏘는 가장 인기 있는 오픈소스 도구
    -   connector.class: io.debezium.connector.postgresql.PostgresConnector
2.  **Delta Lake CDF:** 델타 레이크 테이블 설정에서 enableChangeDataFeed = true를 활성화하면 자체적인 변경 데이터 피드를 사용할 수 있다.

## 3\. 데이터 복제 (Data Replication)

데이터를 있는 그대로 한 장소에서 다른 장소로 복사하는 과정이다. 단순 적재(Loading)보다 데이터의 동질성(기본키, 이벤트 위치 등 메타데이터 보존)을 더 강하게 요구하는 경우가 많다.

**3-1. 패스스루 복제기 (Passthrough Replicator)**

**"환경이 달라도 데이터는 똑같아야 할 때"** 사용한다.

-   **배경:** 개발/스테이징/운영 환경이 나뉘어 있을 때, 운영 환경의 참조 데이터셋을 하위 환경에서도 동일하게 사용해야 버그 탐지가 쉽다.
-   **해결책:** 데이터 변환 없이(Passthrough) 원시 파일이나 레코드를 있는 그대로 복사하는 **EL(Extract-Load)** 잡에 의존한다.
-   **보안 전략:** 운영 환경을 보호하기 위해, 운영 환경에서 하위 환경으로 데이터를 밀어넣는 **푸시(Push) 방식**을 사용하여 프로세스를 통제한다.

**3-2. 변환 복제기 (Transformation Replicator)**

**"데이터는 필요하지만, 민감 정보는 가려야 할 때"** 사용한다.

-   **문제:** 운영 데이터를 테스트용으로 쓰고 싶지만, 개인식별정보(PII)가 포함되어 있어 법적/보안상 그대로 복제할 수 없는 상황이다.
-   **해결책:** 복제 과정 중에 필터링이나 마스킹 같은 **변환 단계**를 추가한다. 단순 복제기보다 훨씬 많은 코드가 필요할 수 있지만, 데이터 보안을 유지하면서도 실데이터 기반의 품질 테스트가 가능해진다.

**1. 주요 구현 방식**  
데이터 축소 (Data Reduction): 불필요하거나 민감한 필드를 아예 제거.  
SQL 활용: SELECT \* EXCEPT (ip, latitude, longitude) 처럼 특정 컬럼을 제외하고 선택.  
PySpark 활용: df.drop('ip', 'latitude', 'longitude') 함수를 사용하여 컬럼을 삭제.  
컬럼 기반 변환: 민감한 필드의 값을 수정.  
  
예시: withColumn을 사용하여 full\_name의 일부만 남기거나 마스킹 처리.  
  
매핑 함수 (Mapping Function): 복잡한 수정 규칙이 필요할 때 사용한다. 스칼라(Scala)의 map 함수 등을 이용해 레코드 단위로 정교한 변환 로직을 적용할 수 있다.  
  
**2. 주의사항 (결과와 단점)**  
데이터셋 손상 위험: 사용자 정의 로직이 추가되므로 패스스루 방식보다 오류 발생 가능성이 높다.  
  
형 변환 문제: JSON/CSV 같은 텍스트 형식은 스키마 정의가 엄격하지 않아 타임스탬프 형식 등이 깨질 수 있다. 가급적 문자열로 정의하여 자동 형 변환 오류를 방지해야 한다.  
  
PII 동적 관리: 현재는 민감하지 않은 정보가 미래에는 개인정보(PII)로 재분류될 수 있으므로, 데이터 거버넌스 도구와 연동하여 변환 로직을 자동화하는 것이 좋다.

주요 기술 구현 예시

1.  **Apache Spark:** sortWithinPartitions('offset') 등을 사용하여 카프카 토픽 복제 시 레코드의 순서를 엄격하게 보장하며 복제할 수 있다.
2.  **Terraform:** aws\_s3\_bucket\_replication\_configuration 리소스를 통해 인프라 수준에서 버킷 복제를 자동화.

## 4\. 데이터 컴팩션

컴팩터 (Compactor)  
실시간 수집 과정에서 발생하는 '작은 파일 문제(Small File Problem)'를 해결하여 읽기 성능을 최적화하는 패턴이다.  
  
1. 문제점  
실시간 스트리밍으로 데이터를 저장하면 수많은 작은 파일이 생성된다.  
파일이 너무 많아지면 메타데이터 조회 오버헤드가 커져, 전체 실행 시간의 70%가 파일 나열(Listing)에 소모될 수 있다.  
  
2. 해결책  
여러 개의 작은 파일을 읽어 하나의 큰 파일로 병합(Merge)하여 전체적인 I/O 비용을 줄인다.  
  
기술별 구현:  
델타 레이크: OPTIMIZE 명령어를 사용하여 파일을 병합힌다.  
아파치 아이스버그: rewrite data files 작업을 수행한다.  
아파치 카프카: 키 기반의 로그 컴팩션을 통해 최신 상태만 유지하며 공간을 최적화한다.  
  
3. 사후 정리 (VACUUM)  
컴팩션 후에도 원본 작은 파일들은 스토리지에 남아 있다.  
VACUUM 명령어를 통해 사용하지 않는 오래된 파일들을 물리적으로 삭제하여 저장 공간을 회수해야 한다.

## 5\. 데이터 준비

**준비 마커 (Readiness Marker)**  
다운스트림(소비자) 팀에게 "데이터셋이 이제 사용 가능한 상태"임을 명시적으로 알려주는 패턴이다.  
  
1. 필요성  
업스트림에서 데이터 적재가 완료되었더라도 품질 검사나 추가 컨텍스트 강화 작업이 끝나지 않았을 수 있다.  
소비자가 불완전한 데이터를 읽어 분석 결과가 틀리는 상황을 방지해야 한다.  
  
2. 구현 방법  
플래그 파일: 데이터 생성이 성공한 후 특정 경로에 \_SUCCESS 같은 빈 파일을 생성한다. 아파치 스파크 등에서 기본적으로 지원하는 방식이다.  
커밋 로그 및 메타데이터: 델타 레이크의 새로운 커밋 로그 등을 확인하여 완료 여부를 판단.  
다음 파티션 감지: 시간 기반 파티셔닝의 경우, 11시 파티션 데이터가 들어오기 시작하면 10시 파티션이 완료된 것으로 간주하는 관례를 활용한다.  
  
3. 고려사항  
지연 데이터(Late Data): 파티션이 닫힌 후에 뒤늦게 도착한 데이터는 무시될 위험이 있으므로, 파티션 갱신 규칙을 컨슈머와 명확히 공유해야 한다.

## 6\. 이벤트 주도 수집 (Event-Driven Ingestion)

정해진 시간에 데이터를 가져오는 '배치(Batch)' 방식과 달리, 데이터가 발생하는 시점에 즉시 반응하여 수집하는 방식이다.

**외부 트리거 (External Trigger)**

데이터가 언제 들어올지 예측하기 어려운 시나리오에서, 데이터 생성을 알리는 이벤트에 반응하여 파이프라인을 실행하는 패턴이다.

**1\. 핵심 개념: 푸시(Push) vs 풀(Pull)**

-   **풀(Pull) 시맨틱:** 컨슈머(수집기)가 새로운 데이터가 있는지 주기적으로 확인(Polling)하는 방식이다. 데이터가 없어도 계속 확인해야 하므로 자원이 낭비될 수 있다.
-   **푸시(Push) 시맨틱:** 프로듀서(생성자)가 데이터 가용성을 컨슈머에게 직접 알린다. 외부 트리거는 이 푸시 방식을 선호하며, 실제 처리할 데이터가 있을 때만 파이프라인을 가동해 효율적이다.

**2\. 외부 트리거의 3단계 작동 원리**

1.  **알림 채널 구독하기:** 외부 시스템(이벤트 프로듀서)과 파이프라인 사이의 연결을 설정한다.
2.  **알림에 반응하기 (Handler):** 이벤트 핸들러가 들어온 이벤트를 분석한다. 특정 조건(예: 특정 테이블에 데이터 생성)에 맞는 이벤트만 필터링하여 처리 여부를 결정한다.
3.  **파이프라인 트리거하기:** 핸들러가 오케스트레이션 도구(Airflow 등)의 API를 호출하여 데이터 수집 워크플로를 시작한다.

**3\. 구현 시 고려사항**

-   **실행 컨텍스트(Metadata) 강화:** 단순히 파이프라인을 실행하는 것뿐만 아니라, '왜 트리거되었는지'에 대한 정보를 함께 전달해야 한다.
    -   전달 정보: 트리거 버전, 알림 Envelope, 처리 시간, 실제 이벤트 발생 시간 등.
    -   효과: 장애 발생 시 원인 조사가 훨씬 쉬워짐
-   **오류 관리 및 복원성:** 이벤트가 없으면 작업도 시작할 수 없으므로 **이벤트 유실 방지**가 최우선.
    -   AWS 람다의 경우 '실패한 이벤트 대상(Dead Letter Queue)' 설정을 통해 인프라 수준에서 복원성을 확보할 수 있다.

**4\. 예제: Airflow + AWS Lambda 조합**

-   **Airflow 설정:** schedule\_interval을 None으로 설정하여 정해진 스케줄 없이 외부 API 호출로만 실행되도록 만든다.
-   **AWS Lambda 핸들러:** S3 등에 새 파일이 업로드되면 람다가 실행되어 Airflow의 API 엔드포인트로 POST 요청을 보낸다. 이때 파일 경로와 같은 컨텍스트를 payload에 담아 전달한다.
