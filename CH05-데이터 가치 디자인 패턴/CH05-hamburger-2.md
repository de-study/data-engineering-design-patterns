## 1\. 메타데이터 데코레이터 (Metadata Decorator)

래퍼 패턴의 하위 개념이자 실무에서 강력하게 쓰이는 **메타데이터 데코레이터 패턴**은 데이터 변환을 수행할 때 레코드 자체에 '언제, 어떤 잡에 의해, 어떤 규칙으로 처리되었는지'를 증명하는 **기술적/비즈니스적 컨텍스트를 주입**하는 기법이다.

-   **문제 상황:** 파이프라인이 고도화되면서 다양한 소스에서 포맷이 제각각인 데이터가 실시간으로 밀려들기 시작한다. 데이터 처리 도중 특정 필드가 계산되었거나 포맷이 변경되었을 때, 다운스트림 분석가나 디버깅 중인 엔지니어가 **"이 레코드가 정확히 몇 번 버전의 배치 잡에 의해, 언제 최종 변경되었는지"** 추적하기가 불가능해진다. 원래 값을 유지하는 것만으로는 데이터의 계보(Lineage)와 신뢰성을 보장하기 어렵다.
-   **해결책:** 레코드의 본질적인 비즈니스 데이터 외에, 파이프라인의 실행 환경 정보(처리 시간, 잡 ID, 코드 버전 등)를 메타데이터 영역으로 정의하여 레코드마다 명시적으로 붙여(Decorate) 저장한다.

#### 1-2. 메타데이터 데코레이션의 4가지 스키마 구조 (상세 분석)

정형 데이터 환경에서 메타데이터 데코레이터를 구현할 때는 스키마를 어떻게 설계하느냐에 따라 4가지 시나리오로 나뉜다. 각각의 장단점이 뚜렷하므로 파이프라인 스토리지의 특성에 맞게 선택해야 한다.

**유형 1: 원시 데이터 평면 구조 + 메타데이터 중첩(Nested) 구조**

-   **구개(구조):** 원본 비즈니스 필드들은 테이블의 최상위(Root) 레벨에 평면하게 두고, 시스템이 추가한 메타데이터 필드들만 metadata struct(...) 같은 하위 객체로 묶어서 저장한다.
-   **특징:** 원본 데이터를 조회할 때 기존 쿼리를 그대로 쓸 수 있어 다운스트림 소비자의 편의성이 높다.

**유형 2: 원시 데이터 중첩 구조 + 메타데이터 평면 구조**

-   **구개(구조):** 유형 1과 반대로, 시스템 메타데이터 컬럼들(processed\_at, job\_id 등)은 최상위 레벨에 배치하고, 원본 데이터 전체를 raw\_data struct(...) 구조체 안으로 격하시켜 넣는 방식이다.
-   **특징:** 파이프라인 모니터링 시스템이나 인프라 팀이 메타데이터를 기반으로 데이터를 파티셔닝하거나 필터링할 때 성능상 매우 유리하다.

**유형 3: 단일 평면(Flat) 구조로 통합**

-   **구개(구조):** 계층 구조를 완전히 배제하고, 원본 필드와 메타데이터 필드를 구별 없이 하나의 평면 테이블에 컬럼으로 나열한다.
-   **특징:** 중첩 구조를 지원하지 않거나 처리가 까다로운 전통적인 RDBMS 환경에서 주로 쓰인다. 단, 필드명이 겹치면 충돌이 날 수 있으므로 메타데이터 필드에 접두사(예: meta\_processed\_at)를 붙여야 한다.

**유형 4: 별도 메타데이터 테이블로 분리 (정규화)**

-   **구개(구조):** 원본 데이터 테이블과 메타데이터 테이블을 완전히 독립시키고, 두 테이블을 고유한 트랜잭션 ID나 레코드 ID 키로 조인(Join)할 수 있게 설계한다.
-   **특징:** 두 데이터셋의 수명 주기(Retention)를 다르게 가져가거나, 저장 공간을 극도로 아껴야 할 때 유용하다. 하지만 조회 시 매번 조인 연산이 발생하므로 성능 트레이드오프를 고려해야 한다.

#### 1-3. 결과 및 디자인 트레이드오프

메타데이터 데코레이터 패턴은 파이프라인의 투명성을 높여주지만, 인프라 비용 측면에서 확실한 고려사항이 존재한다.

-   **데이터 계보(Lineage)와 디버깅의 혁신:** 데이터에 문제가 생겼을 때 버그가 발생한 코드의 버전과 실행 배치를 즉시 역추적할 수 있어 장애 대응 시간이 극적으로 단축된다.
-   **스토리지 크기와 트래픽 증가 제한:** 레코드마다 메타데이터 구조가 복제되므로 전체 데이터셋의 물리적 크기가 커지고 네트워크 트래픽 부담이 늘어난다.
-   **인프라적 완화 전략:** 이 패턴을 적용할 때는 행 전체를 읽지 않고 필요한 컬럼만 선택적으로 읽을 수 있는 컬럼형 데이터 스토리지(Parquet, ORC 파일 포맷 또는 AWS Redshift, GCP BigQuery 등)를 함께 도입하여 트래픽 비효율 문제를 깔끔하게 완화하는 것이 엔지니어링 관례다.

1-4. 메타데이터 데코레이터 구현 코드 예제

PySpark에서의 메타데이터 및 실행 컨텍스트 주입  
잡 버전과 배치 번호 같은 실행 컨텍스트 정보를 구조체(F.struct)로 묶어 원본 레코드와 결합한 뒤, 최종적으로 하나의 JSON 문자열로 직렬화(Envelope)하여 내보내는 코드다.

```
# 예제 5-7: 파이스파크로 메타데이터 래핑하기
from pyspark.sql import functions as F

# 1. 파이프라인의 실행 컨텍스트 정보를 구조체(Struct) 컬럼으로 명시적 생성
visits_w_processing_context = visits.withColumn(
    'processing_context', 
    F.struct(
        F.lit(job_version).alias('job_version'), 
        F.lit(batch_number).alias('batch_version')
    )
)

# 2. 원래 데이터(raw_data)와 기술적 메타데이터를 품은 하나의 고수준 봉투(Envelope)를 형성
visits_to_save = visits_w_processing_context.withColumn(
    'value', 
    F.to_json(F.struct(
        F.col('value').cast('string').alias('raw_data'),
        F.col('processing_context')
    ))
)
```

Spark SQL에서 NAMED\_STRUCT를 이용한 메타데이터 데코레이션 (유형 1)  
원래 필드는 그대로 유지한 채, 계산된 비즈니스 메타데이터(연결 여부, 참조 키 등)를 decorated라는 하위 구조체로 모아서 붙여주는 SQL 연산이다.

```
-- 예제 5-8: NAMED_STRUCT를 활용한 데코레이션
SELECT *, 
       NAMED_STRUCT(
           'is_connected', CASE WHEN context.user.connected_since IS NULL THEN false ELSE true END,
           'page_referral_key', CONCAT_WS('-', page, context.referral)
       ) AS decorated 
FROM input_visits;
```

Spark SQL에서 원시 값을 하위 구조체로 격하하는 데코레이션 (유형 2)  
계산된 추적 메타데이터들을 최상위 컬럼으로 올리고, 원래의 원시 레코드 전체를 raw라는 구조체 안으로 밀어 넣는 방식이다.

```
-- 예제 5-9: 원시 레코드를 하위 구조체로 격하하여 분리하기
SELECT 
    CASE WHEN context.user.connected_since IS NULL THEN false ELSE true END AS is_connected,
    CONCAT_WS('-', page, context.referral) AS page_referral_key,
    STRUCT(visit_id, event_time, user_id, page, context) AS raw
FROM input_visits;
```

주의: 이처럼 원시 값과 가공된 메타데이터를 구조적으로 완전히 격리해주지 않으면, 최종 사용자가 어떤 데이터가 원본이고 어떤 데이터가 시스템에 의해 계산된 것인지 출처를 혼동하여 잘못된 비즈니스 의사결정을 내릴 위험이 있다.

## 2\. 분산 집계기 (Distributed Aggregator) 패턴

**'분산 집계기'** 패턴은 분산 클러스터 인프라 전체의 구조를 활용해 부하를 물리적으로 쪼개는 아키텍처적 솔루션이다.

-   **문제 상황:** 특정 마케팅 캠페인이나 블랙 프라이데이 이벤트로 인해 특정 키(예: 핫스팟 상품 ID, 특정 국가 코드)에 수억 건의 데이터가 집중되는 **'데이터 스큐(Data Skew, 데이터 편향)'** 현상이 발생한다. 로컬 집계기를 적용하더라도, 결국 최종 글로벌 집계를 위해 해당 키의 모든 데이터가 네트워크를 타고 단 하나의 특정 리듀서 노드로 몰리게 된다. 이로 인해 해당 노드의 메모리가 고갈되어 OOM(Out of Memory)으로 파이프라인이 크래시되거나, 그 노드의 처리가 끝날 때까지 전체 잡이 마비되는 병목이 발생한다.
-   **해결책:** 하나의 핫 키(Hot Key)를 여러 개의 서브 키로 쪼개어 여러 노드가 나누어 처리하도록 유도하는 **2단계 분산 집계(Two-Phase Distributed Aggregation)** 구조를 아키텍처에 강제한다.

**2-1. 작동 원리: 2단계 분산 집계 아키텍처**

분산 집계기 패턴은 대규모 스큐 문제를 해결하기 위해 '솔팅(Salting, 소금치기)' 기법을 활용한 2단계 파이프라인을 구성한다.

1단계: 솔팅(Salting) 기반의 1차 분산 집계

-   **메커니즘:** 유입되는 레코드의 원래 그룹화 키에 무작위 임의의 숫자(Salt)를 접두사나 접미사로 붙인다. 예를 들어, item\_A라는 키에 1부터 4까지의 랜덤 숫자를 붙여 item\_A\_1, item\_A\_2, item\_A\_3, item\_A\_4라는 4개의 서로 다른 서브 키로 변환한다.
-   **효과:** 분산 엔진은 이 서브 키들을 완전히 다른 키로 인식하므로, 단 하나의 리듀서가 아닌 클러스터 내의 서로 다른 4개의 노드로 데이터를 균등하게 분산(스캐터)시켜 1차 집계를 수행한다.

2단계: 솔트 제거 후 최종 2차 글로벌 집계

-   **메커니즘:** 1차 집계가 끝나 부피가 급격히 줄어든 서브 키 결과물에서 뒤에 붙였던 랜덤 숫자(Salt)를 다시 떼어낸다. 즉, item\_A\_1 ~ item\_A\_4의 결과물들을 다시 원래의 item\_A라는 키로 통일한다.
-   **효과:** 이미 1차 단계에서 데이터가 대폭 압축되었기 때문에, 최종 리듀서 노드로 데이터가 다시 모이더라도(게더) 노드가 감당할 수 있는 아주 가벼운 수준의 연산만 수행하고 최종 집계를 안전하게 완료한다.

**2-2. 결과 및 디자인 트레이드오프**

분산 집계기는 테라바이트(TB)나 페타바이트(PB) 수준의 초대형 데이터 스큐를 극복할 수 있는 유일한 해결책에 가깝지만, 아키텍처적 복잡도가 올라간다.

-   **OOM 장애의 근본적 방어:** 특정 워커 노드 하나가 독박을 써서 터지는 현상을 완벽히 방지하여, 대규모 트래픽 인입 시에도 파이프라인의 예측 가능성과 안정성이 보장된다.
-   **파이프라인 단계 증가와 관리 오버헤드:** 하나의 단순 GROUP BY 연산이 1차 집계, 솔트 제거, 2차 집계라는 최소 2~3단계의 복잡한 실행 계획(DAG)으로 쪼개지므로 관리 포인트가 늘어난다.
-   **적절한 솔트 범위 설정의 어려움:** 랜덤 숫자의 범위(예: 1~N)에서 N의 크기를 너무 작게 잡으면 스큐 방어 효과가 떨어지고, 반대로 너무 크게 잡으면 오히려 불필요한 서브 파티션이 대량 양산되어 네트워크 및 메타데이터 관리 부하가 증가한다. 비즈니스 데이터의 편향도를 모니터링하여 최적의 밸런스를 찾아야 한다.

2-3. 예제

핫 키 문제를 방지하기 위해 1부터 9까지의 랜덤 값을 활용해 데이터를 분산시켜 1차 집계를 한 뒤, 2차로 최종 취합하는 프로그래밍 기법이다.

```
-- 예제 5-19: Spark SQL을 활용한 2단계 분산 집계기 패턴 (Salting 기법)

-- [1단계] 원래 키에 랜덤 값(0~9)을 결합하여 데이터를 쪼갠 후 1차 집계 수행
WITH first_phase_agg AS (
    SELECT 
        page,
        -- 원래 page 키 뒤에 0에서 9 사이의 랜덤 숫자를 붙여 서브 키 형성
        CONCAT(page, '_', CAST(FLOOR(RAND() * 10) AS STRING)) AS salted_page,
        COUNT(*) AS intermediate_count
    FROM 
        dedp.raw_visits
    WHERE 
        visit_date = '${execution_date}'
    GROUP BY 
        page,
        CONCAT(page, '_', CAST(FLOOR(RAND() * 10) AS STRING))
),

-- [2단계] 서브 키에서 다시 랜덤 숫자를 분리하여 원본 키로 복원
salted_removed AS (
    SELECT 
        -- 문자열 파싱을 통해 뒤의 '_숫자'를 제거하고 원본 page 키 추출
        SPLIT(salted_page, '_')[0] AS original_page,
        intermediate_count
    FROM 
        first_phase_agg
)

-- [3단계] 압축된 중간 결과들을 바탕으로 최종 글로벌 집계 마무리
SELECT 
    original_page AS page,
    SUM(intermediate_count) AS total_visits
FROM 
    salted_removed
GROUP BY 
    original_page;
```

## 3\. 로컬 집계기 (Local Aggregator) 패턴

앞서 다룬 집계기(Aggregator) 패턴은 데이터의 부하를 줄여주는 훌륭한 도구이지만, 분산 컴퓨팅 환경에서 GROUP BY 연산을 수행할 때 노드 간에 엄청난 양의 데이터를 전송하는 '네트워크 셔플(Shuffle)'을 유발한다는 치명적인 숙제를 안고 있다.

-   **문제 상황:** 특정 핫스팟 페이지(예: 이벤트 메인 페이지)의 방문 수가 폭발적으로 증가하면서 실시간 집계 파이프라인이 급격히 느려지기 시작한다. 분산된 개별 워커 노드들이 각자 수집한 대량의 원시 레코드를 네트워크를 통해 단일 마스터/리듀서 노드로 그대로 전송하기 때문에, 네트워크 대역폭이 고갈되고 특정 노드에 부하가 집중되는 병목 현상이 발생한다.
-   **해결책:** 데이터를 네트워크로 전송하기 전에, 각 워커 노드의 로컬 메모리 상에서 먼저 1차적으로 미니 집계(중간 요약)를 수행한 뒤 그 결과만 네트워크로 내보내는 **로컬 집계기 패턴**을 적용한다.

**3-1. 작동 원리: 맵리듀스의 컴바이너(Combiner) 시맨틱**

로컬 집계기 패턴은 빅데이터 분산 처리의 고전적인 개념인 맵리듀스(MapReduce)의 '컴바이너(Combiner)'나 아파치 스파크의 reduceByKey 메커니즘과 동일하게 작동한다.

-   **1단계: 로컬 버퍼링 및 1차 집계 (Map Side):** 각 워커 노드는 유입되는 데이터를 즉시 네트워크로 던지지 않고, 로컬 메모리 버퍼(윈도우)에 아주 잠깐 동안(예: 몇 초 또는 수백 건 단위) 모은다. 그 후 로컬에서 동일한 키를 가진 레코드들을 먼저 합산(COUNT, SUM 등)한다.
-   **2단계: 최적화된 셔플 (Shuffle):** 로컬 집계를 거치면서 데이터의 로우(Row) 수가 극적으로 줄어든다. 예를 들어, 10,000건의 원시 방문 레코드가 로컬에서 (page\_A, 10000)이라는 단 1건의 중간 집계 레코드로 압축되어 네트워크를 탄다.
-   **3단계: 최종 글로벌 집계 (Reduce Side):** 마스터 노드(또는 리듀서)는 각 워커 노드가 보내온 1차 집계 결과물들을 취합하여 최종 결과를 산출한다.

**3-2. 결과 및 디자인 트레이드오프**

로컬 집계기는 대규모 분산 파이프라인의 생존을 위해 필수적인 패턴이지만, 모든 상황에서 만능은 아니다.

-   **네트워크 대역폭 절감 및 데이터 스큐(Skew) 완화:** 네트워크를 통과하는 데이터의 절대적인 부피가 줄어들기 때문에 클러스터 전체의 데이터 전송 효율이 극대화된다. 특히 특정 키에 데이터가 몰리는 '데이터 스큐' 현상이 발생했을 때 리듀서 노드가 터지는 문제를 완화해 준다.
-   **추가적인 로컬 메모리 소모:** 각 워커 노드가 중간 집계 결과를 메모리에 들고 있어야 하므로, 워커 노드의 메모리 압박이 약간 증가한다.
-   **고유 키(Cardinality) 크기에 따른 효율성 트레이드오프:** 만약 그룹화 기준 키의 카디널리티가 너무 커서(예: 전 세계 사용자 UUID) 로컬 노드 내에서 중복되는 키가 거의 없다면, 로컬 집계를 수행하더라도 데이터가 압축되지 않는다. 이 경우 오히려 버퍼링과 1차 연산으로 인한 CPU 자원 낭비 및 지연 시간(Latency) 증가라는 역효과를 초래할 수 있다.

**3-3. 로컬 집계기 패턴 구현 코드 예제**

PySpark에서 reduceByKey를 활용한 로컬 집계 최적화

아파치 스파크에서 groupByKey().mapValues(sum) 구조를 사용하면 데이터 압축 없이 모든 원시 레코드가 셔플을 타게 된다. 반면, 아래와 같이 reduceByKey를 사용하면 스파크 엔진이 내부적으로 로컬 집계기(Map-side combine)를 자동 적용하여 네트워크 트래픽을 최적화한다.

```
# 예제 5-16: 파이스파크 reduceByKey를 통한 로컬 집계기 구현
# 1. 원시 RDD 데이터를 (키, 값) 쌍으로 변환 -> (page, 1)
page_pairs = visits_rdd.map(lambda x: (x.page, 1))

# 2. reduceByKey를 호출하여 로컬 노드에서 먼저 합산한 후 셔플 수행
# 이 연산은 내부적으로 로컬 집계기 패턴을 강제하므로 대규모 데이터 처리 시 성능이 압도적이다.
total_page_counts = page_pairs.reduceByKey(lambda a, b: a + b)
```

일반 스트리밍 애플리케이션에서의 인메모리 로컬 버퍼 집계  
프레임워크의 내장 기능 도움 없이, 메모리 딕셔너리와 카운터 임계치(FLUSH\_THRESHOLD)를 활용해 로컬에서 데이터를 먼저 압축한 뒤 브로커(Kafka 등)로 발행하는 저수준 구현 구조다.

```
# 예제 5-17: 저수준 인메모리 로컬 집계기 의사코드
class LocalAggregator:
    FLUSH_THRESHOLD = 500  # 로컬 버퍼에 모을 최대 레코드 수

    def __init__(self):
        self.local_buffer = {}  # 로컬 메모리 딕셔너리
        self.record_count = 0

    def process_event(self, page):
        # 1. 로컬 메모리에서 즉시 1차 집계 수행
        self.local_buffer[page] = self.local_buffer.get(page, 0) + 1
        self.record_count += 1

        # 2. 임계치에 도달하면 중간 집계 결과를 다운스트림으로 내보내고 버퍼 비우기
        if self.record_count >= self.FLUSH_THRESHOLD:
            self._flush_to_network()

    def _flush_to_network(self):
        for page, count in self.local_buffer.items():
            # 원시 데이터 500개를 보내는 대신, 압축된 중간 집계본만 네트워크로 전송
            send_to_downstream(page, count)
        self.local_buffer.clear()
        self.record_count = 0
```

## 4\. 증분 세션화 처리기 (Incremental Sessionizer) 패턴

앞서 다룬 일반 세션화기 패턴은 전체 데이터셋을 펼쳐놓고 연산하거나(배치), 사용자의 모든 상태를 메모리에 유지해야 하므로(스트리밍) 데이터 규모가 커질수록 자원 소모가 기하급수적으로 늘어나는 한계가 있다. 특히 배치 파이프라인에서 매일 세션을 처음부터 다시 계산하는 방식은 엄청난 컴퓨팅 낭비를 초래한다.

-   **문제 상황:** 매일 수억 건의 방문 로그가 쌓이는 서비스에서 일별 배치로 세션화 작업을 수행하고 있다. 밤 11시 50분에 접속해 다음 날 새벽 1시까지 활동하는 '날짜 경계에 걸친 세션'을 정확히 묶기 위해, 매번 과거 며칠 분량의 데이터를 통째로 읽어와 재연산(Full Re-computation)해야 하는 비효율이 발생한다. 데이터 볼륨이 커짐에 따라 배치 처리 시간이 늘어나고 인프라 비용이 감당하기 어려운 수준으로 치솟는다.
-   **해결책:** 과거의 완성된 세션 결과물은 그대로 두고, **'아직 닫히지 않은(진행 중인) 활성 세션' 정보만 별도의 상태 저장소에 가볍게 유지**하면서 새로 유입된 증분(Incremental) 데이터와 결합해 세션을 갱신하는 증분 세션화 처리기 패턴을 적용한다.

**4-1. 작동 메커니즘: 활성 세션 상태 테이블의 활용**

증분 세션화 처리기는 모든 데이터를 다시 읽는 대신, 과거의 유산과 현재의 데이터만 매핑하는 영리한 3단계 파이프라인 구조를 가진다.

1.  **활성 세션 추출 (Active Sessions Read):** 이전 배치 주기까지 종료되지 않고 '진행 중' 상태로 분류된 세션 목록(최근 이벤트 타임스탬프, 누적 방문 수 등)을 마스터 상태 테이블에서 읽어온다.
2.  **증분 데이터 조인 및 갱신:** 새로 유입된 시간대의 증분 로그셋을 활성 세션 목록과 user\_id 기준으로 조인한다.
    -   새 이벤트가 기존 활성 세션의 무활동 임계치(예: 30분) 이내에 들어오면 기존 세션 ID를 그대로 이어받고 타임스탬프를 갱신한다.
    -   임계치를 넘겼다면 기존 세션은 '최종 종료'로 마크하고, 새 이벤트를 시작점으로 하는 신규 세션 ID를 발급한다.
3.  **상태 업데이트 및 저장 (Write Back):** 연산이 완료되면 종료된 세션은 최종 분석 레이어로 내보내고, 여전히 진행 중인 세션들만 상태 테이블에 다시 덮어써서 다음 증분 배치를 준비한다.

**4-2. 결과 및 디자인 트레이드오프**

이 패턴은 빅데이터 아키텍처에서 컴퓨팅 자원을 드라마틱하게 아껴주지만, 설계 구조가 복잡해지는 대가가 따른다.

-   **컴퓨팅 비용의 분동(혁신적 절감):** 매번 대규모 전체 데이터를 정렬하고 셔플할 필요 없이, 오직 당일 유입된 증분 데이터와 소량의 활성 세션 테이블만 조인하므로 IO 비용과 CPU 부하가 극적으로 감소한다.
-   **상태 관리의 복잡성:** 진행 중인 세션을 저장하고 관리할 별도의 지속성 스토리지(HBase, DynamoDB 또는 데이터 레이크의 Delta/Iceberg 테이블)가 필요하며, 상태 데이터가 오염되거나 유실되었을 때를 대비한 복구(Recovery) 메커니즘을 별도로 구현해야 한다.
-   **지연 데이터(Late Data) 처리의 한계:** 만약 이틀 전 발생한 이벤트가 시스템 장애로 뒤늦게 증분 파이프라인에 유입되면, 이미 상태 테이블에서 만료되어 최종 종료 처리된 세션과 결합하지 못하고 유실되거나 파편화될 수 있다. 따라서 비즈니스 허용 범위에 맞춰 활성 세션의 보존 기간(Retention Window)을 넉넉하게 설정해야 한다.

**4-3. 증분 세션화 처리기 구현 코드 예제**

Spark SQL 환경에서의 증분 세션 병합 및 업데이트 (배치 관점)  
과거의 활성 세션 테이블(active\_sessions)과 새로 유입된 증분 로그(incremental\_visits)를 결합하여 세션을 연장하거나 새로 생성하는 구조의 쿼리다.

```
-- 예제 5-18: Spark SQL을 활용한 증분 세션화 처리기 패턴
WITH combined_data AS (
    -- 1. 진행 중인 활성 세션 정보와 신규 증분 데이터를 하나로 통합
    SELECT user_id, last_event_time AS event_time, session_id, 'ACTIVE' AS source FROM dedp.active_sessions
    UNION ALL
    SELECT user_id, event_time, NULL AS session_id, 'INCREMENTAL' AS source FROM dedp.raw_incremental_visits
),
ranked_events AS (
    SELECT *,
           -- 이전 이벤트와의 공백 계산
           EXTRACT(EPOCH FROM event_time) - 
           LAG(EXTRACT(EPOCH FROM event_time)) OVER (PARTITION BY user_id ORDER BY event_time ASC) AS gap_seconds
    FROM combined_data
)
-- 2. 최종 쿼리 단계에서 gap_seconds가 1800초(30분)를 넘으면 새로운 세션 ID를 발급하고, 
-- 그렇지 않으면 기존 활성 세션의 session_id를 유지하도록 로직을 전개하여 상태 테이블을 갱신한다.
SELECT 
    user_id,
    event_time,
    COALESCE(session_id, CONCAT(user_id, '_', CAST(EXTRACT(EPOCH FROM event_time) AS STRING))) AS final_session_id
FROM ranked_events;
```
